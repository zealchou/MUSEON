Phase 2：License & Subscription（順序 4–9）

  • 產出：POST /api/v1/licenses/:id/pair-code

  • 產出：subscriptions 表 + ACTIVE/GRACE/SUSPENDED 轉換邏輯

  • 產出：@Cron 排程，每 5 分鐘檢查
  • 規則：冪等、不重複觸發 side effects（ref：STATE_RULES §9）

SYSTEM_SCOPE

包含範圍（In Scope）

本文件定義 MUC MVP 第一階段必須交付的系統範圍。

本階段範圍以「收費閉環 + 設備授權閉環 + 最低防破解基線」為主，不追求完整 SaaS 功能，而是先確保產品能上線、能收費、能控管設備合法使用權。

1. 帳號與身分驗證

  • 使用者註冊
  • 使用者登入 / 登出
  • 基本帳號狀態管理
  • 使用者可登入前台查看自己的授權與設備資訊

2. Billing Profile / 付款資料

  • 使用者建立與維護付款資料
  • 平台可查閱付款資料以進行人工對帳與開通
  • 作為人工收款、續期、通知與後續發票作業的基礎資料

3. License / Serial 管理

  • 平台建立序號（可指定方案）
  • 使用者查詢自己持有的序號
  • 平台查看序號綁定設備
  • 平台可啟用 / 停用 / 作廢序號
  • 明確以序號作為計費單位，而非以帳號作為計費單位
  • License 綁定 Plan（方案決定金額、期限、信任點配額）

3.5 方案管理（Plan）

  • 方案分兩類：訂閱（含設定費 NT$24,000）與續約（純月費 NT$6,000/月）
  • 訂閱方案：A(設定+2月) / B(設定+6月) / C(設定+12月) / D(客製)
  • 續約方案：1月 / 6月 / 12月
  • 每個方案可設定一種優惠（送月 / 送點 / 折扣，三選一不可疊加）
  • 客製方案（訂閱 D）金額和期限由 Admin 每次指定
  • Portal 顯示邏輯：新用戶看訂閱方案，既有用戶看續約方案
  • 信任點：300 點/月（所有標準方案），超額 USD 0.1/點

4. Subscription 週期與狀態管理

  • 每個序號各自擁有獨立訂閱週期
  • 週期起算以平台確認收款日（Asia/Taipei）為準
  • 週期長度依方案 duration_months 決定（+1/+2/+6/+12 個月）
  • 到期後進入 GRACE
  • 寬限結束後進入 SUSPENDED
  • 平台確認收款後可由任意狀態恢復為 ACTIVE
  • 狀態需支援 ACTIVE / GRACE / SUSPENDED

5. 人工收款與續期

  • 平台建立人工收款紀錄
  • 平台確認收款後立即延長對應 Subscription 週期
  • 支援 payment_status：DRAFT / CONFIRMED / VOID
  • 收款確認後可觸發通知與後續營運動作

6. Device Pairing / 設備綁定

  • 使用者在前台產生一次性配對碼
  • Agent 以配對碼完成首次綁定
  • 平台建立正式 device record
  • 成功配對後僅回傳一次 device token 明文
  • 一個序號同時只允許一台 ACTIVE 設備
  • 若已有 ACTIVE 設備，需先 revoke 舊設備或明確執行取代流程

7. Device Token / Entitlement

  • Agent 以 device token 定期輪詢授權
  • Token 只存 hash，不存明文
  • Token 可被 revoke
  • Entitlement 必須回傳可驗章之授權結果
  • Agent 端必須驗證簽章、device_id、一致性與有效時間
  • Entitlement 結果需回傳 allow_new_jobs、subscription_status、grace_until 等關鍵資訊

8. Pre-Job Check / 接單前檢查

  • Agent 每次接新任務前必須再次檢查授權
  • 可透過 pre-job check API 或短效 job grant 實現
  • 若授權失效、簽章無效、device mismatch 或 response 過期，則不得接新任務

9. Device Revocation / 換機與停用

  • 平台或使用者可主動停用設備
  • 設備停用後 token 必須立即失效
  • 換機時新設備必須重新配對，不可直接複製既有 token 生效
  • 支援 device_status：ACTIVE / REVOKED / REPLACED

10. 告警中心與通知

  • 收款成功 / 失敗事件
  • 到期 / 寬限 / 停用事件
  • 設備異常事件
  • entitlement 驗證失敗事件
  • fingerprint mismatch、nonce replay、token 異常等風險事件
  • 管理員可 ACK / RESOLVE 告警

11. 稽核紀錄

  • 關鍵操作需寫入 append-only audit log
  • 至少涵蓋：

- 收款確認

- 停用設備

- 重綁設備

- 修改到期日

- 修改付款資料

- pair code 產生 / 使用 / 失敗

- token 發行 / revoke / rotation

- entitlement 驗證失敗

  • 不提供刪除 audit log API

12. 最低安全基線

本階段至少納入以下安全要求：

1. 序號不能直接當授權憑證

2. pair code 一次性、短時效、只存 hash

3. device token 高熵亂數、只存 hash、可 revoke

4. token 綁 device_id 與 fingerprint 檢查

5. entitlement response 必須簽章

6. Agent 端必須內建公鑰驗章

7. 每次接新任務前必查授權或 grant

8. 離線超過 15 分鐘後停止接新任務

主要使用情境（BDD 前置）

情境 1：新用戶開通

1. 使用者註冊帳號

2. 使用者建立付款資料

3. 平台建立第一個序號

4. 平台確認收款

5. Subscription 生效

6. 使用者登入前台看到可用狀態

情境 2：設備首次配對

1. 使用者在設備中心新增設備

2. 系統產生一次性配對碼

3. Agent 輸入配對碼

4. 平台完成設備綁定

5. 平台回傳 device token

6. Agent 以 token 定期查授權

情境 3：正常授權輪詢

1. Agent 定期呼叫 entitlement API

2. 平台驗證 token、device_id、fingerprint、nonce、timestamp

3. 平台回傳短效且可驗章的授權結果

4. Agent 驗章成功後更新本地授權狀態

情境 4：接新任務前檢查

1. Agent 在接新任務前呼叫 pre-job check

2. 平台再次驗證設備與訂閱狀態

3. 若回傳有效授權 / job grant，則允許接新任務

4. 若授權失效或驗章失敗，則停止接新任務

情境 5：續費

1. 使用者收到即將到期提醒

2. 客戶付款

3. 平台人工確認收款

4. 系統更新訂閱期限

5. Agent 在下次輪詢或接單前取得新狀態

情境 6：到期與停用

1. Subscription 到期

2. 狀態轉為 GRACE

3. 寬限結束仍未付款，轉為 SUSPENDED

4. Agent 取得狀態後停止接新任務

5. 已在跑的任務可完成，但不允許再接新工作

情境 7：設備停用 / 換機

1. 平台或使用者主動停用舊設備

2. 舊 token 立即失效

3. 新設備重新產生配對流程

4. 新設備取得新 token 後才可恢復授權

情境 8：異常與風控

1. 發生 fingerprint mismatch、nonce replay、簽章驗證失敗或 token 異常使用

2. 系統建立告警

3. 依風險等級採取僅告警、要求重配、暫停接新任務、直接 revoke token 等措施

整合邊界（外部系統 / API）

1. Agent / Device API

MVP 需提供至少以下裝置側 API：

  • POST /api/device/pair
  • GET /api/device/entitlement
  • POST /api/device/check-before-job

用途如下：

  • 配對與註冊設備
  • 取得短效且可驗章的授權資訊
  • 在接新任務前再次確認授權狀態

2. Admin API / 後台操作 API

MVP 需提供至少以下平台端能力：

  • 建立與管理序號
  • 建立與確認收款
  • 查看與停用設備
  • 查看告警與稽核紀錄
  • POST /admin/devices/{id}/revoke

3. 排程 / Reconciler

平台需具備背景排程能力，用於：

  • ACTIVE → GRACE 狀態校正
  • GRACE → SUSPENDED 狀態校正
  • 通知觸發
  • 例行風控與清理任務

4. Notification / Invoice Worker

可先保留為平台內部模組或後續 worker 邊界：

  • 收款確認後通知
  • 到期 / 寬限 / 停用通知
  • 發票後續任務掛點
  • 不要求第一階段完成完整外部串接

5. 金流邊界

  • 第一階段不串接外部 payment gateway
  • 收款來源可為人工匯款或人工對帳
  • 系統只需支援人工確認與狀態更新，不處理自動扣款

6. 安全邊界

  • Agent 不可信任任意可編輯 license server URL
  • TLS 為必要但非唯一信任來源
  • 平台需提供可驗章 response
  • Agent 需內建平台公鑰並在本地驗章
  • 可先不納入深度 anti-debug / anti-hook，但須保留後續擴充空間
