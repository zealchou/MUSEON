# 施工前強制檢查協議（Pre-Flight Checklist）

> **任何程式碼修改前，必須依序完成。違反此協議是「修A壞B」的根因。**

## 核心清單

-1. □ **Skill 定義優先讀（涉及 Skill 時必做）**：
   - 任務涉及任何 Skill？→ **先讀** `~/.claude/skills/{skill_name}/SKILL.md`，再動手
   - 憑印象做 = 做錯。Skill 的輸出格式、必要章節、分析層數以定義為準，不以過去案例為準
   - 不確定 skill 名稱 → `ls ~/.claude/skills/` 找到後再讀

0. □ **減法審查（最優先）**：在想「要加什麼」之前，先回答：
   - 能不能**刪掉**產生這個問題的原因，而不是在下游加防禦？
   - 有沒有兩套邏輯在做同一件事？→ 刪掉一套，不是加協調層
   - 這個修復會讓系統**更簡單**還是**更複雜**？
   - 如果答案是「更複雜」，必須向使用者說明為什麼減法不可行

1. □ 讀 `docs/joint-map.md`：我要改的模組碰了哪些共享狀態？
2. □ 讀 `docs/blast-radius.md`：改了會影響哪些其他模組？扇入多少？
3. □ 列出「會變的」和「絕對不能變的」
4. □ 如果影響 ≥ 2 個模組，**必須先回報使用者確認才能動手**
5. □ 修完後驗證「不能變的」沒變（跑相關測試）
6. □ 如果改了共享狀態的讀寫格式，同步更新 `docs/joint-map.md`
7. □ 如果改了模組介面（函數簽名、import 關係），同步更新 `docs/blast-radius.md`

## 資料搜尋紀律（Data Retrieval Discipline）

> **在 MUSEON 中搜尋任何資料前，必須先讀藍圖定位。盲目 find/grep 是浪費時間的根因。**

1. □ 先讀 `docs/persistence-contract.md`（水電圖）：資料存在哪個引擎、什麼路徑？
2. □ 再讀 `docs/memory-router.md`（郵路圖）：洞見/記憶的流向是什麼？
3. □ 如果涉及模組關係，讀 `docs/system-topology.md`（神經圖）
4. □ 鎖定位置後才開始讀檔，**不做無目的的 find/grep**

## 行動前紀律（Before-Action Discipline）

> **遇到看似受限的任務，先窮盡替代路徑，不可直接回覆「不可能」或「做不到」。**

1. □ 工具有限制？先問：資料有沒有可能在其他地方？
2. □ 窮盡本地檔案系統、日誌、快取、資料庫等替代來源
3. □ 只有在確實查過所有合理路徑後，才告知使用者限制
4. □ 即使最終真的做不到，也要說明「查了哪些地方、為什麼都沒有」

---

## 施工後驗證紀律（Post-Implementation Verification）

> **手動測試通過 ≠ 系統會執行。「我呼叫函數得到正確結果」不代表「系統排程會呼叫這個函數」。**

1. □ 新增 Nightly step？確認 **`_FULL_STEPS` 列表**包含新步驟 ID
2. □ 新增事件處理？確認**觸發源存在**且路徑匹配（讀寫入端代碼確認實際路徑，不用猜的）
3. □ 新增消費端？確認**生產端存在**且有資料（heuristics.json 不存在 = 消費端是空殼）
4. □ 降低門檻？確認**計數器有遞增機制**（record_success() 沒人呼叫 = 門檻再低也無用）
5. □ 重大修改完成後，**spawn 獨立審計 subagent** 做反向 BDD 驗證——自己驗自己有盲點
