# 劇本簡化協議（Screenplay Simplification Protocol）

> **來源**：2026-03-25 Luo 的舞蹈劇本簡化需求
> **問題**：學校臨時要求從 5 分鐘縮到 3.5 分鐘，AI 硬壓導致「2 倍速感」
> **解法**：不是單純砍字，而是重新設計「節奏層次」

---

## 核心原則

### 1. 時長控制演算法

**輸入**：
- 總演出時間 T_total（秒）
- 舞蹈段落數 n_dance
- 舞蹈時長 T_dance（秒）

**計算**：
```
T_dialogue = T_total - T_dance
T_dialogue_per_scene = T_dialogue / n_scenes
```

**例子**（Luo 的案例）：
- T_total = 210 秒（3.5 分鐘）
- T_dance = 120 秒（2 分鐘：50 秒 Solo + 70 秒融合）
- T_dialogue = 90 秒
- n_scenes = 4（開場、場景一、場景二+三、結尾）
- T_dialogue_per_scene ≈ 22-25 秒

**規則**：
- 開場 ≤ 15 秒（不超過 3 句）
- 每個轉場 ≤ 35 秒
- 結尾 ≤ 15 秒

---

### 2. 節奏層次（Rhythm Architecture）

不同於「平均刪減」，應該用**節奏的加速與遞進**：

| 層次 | 策略 | 例子 |
|------|------|------|
| **開場** | 極簡發問 | 「拉丁舞誰最經典？」 |
| **衝突升級** | 短句對話 + 重複詞 | 「太慢！」「我阿嬤都比你快！」 |
| **群體加速** | 多人輪流反駁，每人 1-2 句 | 鬥牛 → 恰恰 → 倫巴 → 森巴 |
| **轉場觸發** | 決定性台詞 | 「那就比啦！」 |
| **舞蹈緩衝** | 50-70 秒舞蹈，讓觀眾喘氣 | 視舞蹈難度調整 |
| **反轉收束** | 感情線反轉 | 「每種舞都很特別」 |

---

### 3. 多人對白平衡（Dialogue Distribution）

**問題**：一個人講超過 40 字會「拖戲」
**解法**：每人每次 ≤ 20 字，輪流說

**計算**：
```
characters = 4
avg_words_per_turn = 15-20 字
turns_per_scene = (T_dialogue_per_scene / avg_words_per_minute) * 60
                = (25 秒 / 150 字/分鐘) ≈ 6-7 輪
```

**分配策略**：
- 開場：2 人說
- 衝突場景：4 人輪流，每人 1-2 次
- 關鍵反轉：1 人（製造驚喜）
- 結尾：全體齊聲

---

### 4. 舞蹈融合邏輯（Dance Integration Logic）

舞蹈不只是「劇本後面插舞」，而是**敘事的延續**：

| 階段 | 功能 | 時長 | 內容 |
|------|------|------|------|
| 第一段 Solo | 各自展示 | 50 秒 | 四人輪流秀自己的舞種特色 |
| 第二段融合 | 共識體現 | 70 秒 | 融合多種舞蹈元素（捷舞節奏） |
| 定格 | 視覺結尾 | 5 秒 | 所有人同時定格 Pose |

**銜接口白**：
- 進 Solo 前：「那就比啦！」（決定點）
- Solo 中間：可穿插掌聲 / 觀眾反應聲
- 進融合前：「大家掌聲都一樣耶！」（轉折點）
- 融合中：無台詞，純舞蹈

## 執行流程（工作流）

### Phase 1: 需求提取（Requirement Extraction）

收集：
- [ ] 總演出時間（分秒）
- [ ] 舞蹈段落數 + 每段時長
- [ ] 演出人數 + 角色名稱
- [ ] 核心笑點清單（必須保留）
- [ ] 小演員已排練過的台詞句子（不可大改）

### Phase 2: 骨架設計（Skeleton Design）

設計：
- [ ] 場景數 + 每場時長預算
- [ ] 節奏曲線（圖表：橫軸=時間，縱軸=節奏快度）
- [ ] 多人對白的輪次分配
- [ ] 舞蹈銜接點 (3 個：進 Solo、離 Solo、定格)

### Phase 3: 台詞精簡（Dialogue Compression）

規則：
- [ ] 保留原台詞的「關鍵詞」
- [ ] 刪除「修飾語 + 重複語」
- [ ] 改「長獨白」為「短對話」
- [ ] 每句 ≤ 20 字，每段 ≤ 25 秒

### Phase 4: 節奏驗證（Rhythm Verification）

檢查清單：
- [ ] 開場有沒有卡頓？（應 < 15 秒）
- [ ] 衝突升級有沒有感覺「快起來」？（應呈加速感）
- [ ] 進舞蹈的轉場自然嗎？（應「決定→行動」，無突兀）
- [ ] 舞蹈後反轉有沒有「呼」一聲感？（應有感人的轉折）
- [ ] 朗讀節奏對嗎？（用正常說話速度讀一遍，應是平穩遞進，不卡不快)

### Phase 5: 排練測試（Rehearsal Test）

驗證：
- [ ] 小演員能快速背下新台詞嗎？（如果超過 3 次排練仍卡詞，說明台詞複雜度過高）
- [ ] 舞蹈銜接順不順？（進舞蹈前有 2 秒的「準備呼吸」嗎？）
- [ ] 觀眾反應如何？（問家長：有沒有感覺卡頓？笑點有沒有出現？）

## 品質指標（Quality Metrics）

### 定量指標

| 指標 | 目標 | 測量方式 |
|------|------|--------|
| 台詞總字數 | 目標 ± 10% | 計算去空格字數 |
| 每句平均字數 | 15-20 字 | 台詞總字數 / 句數 |
| 平均朗讀速度 | 150-180 字/分鐘 | 朗讀全劇本，計時 |
| 每人說話時間 | 均勻分布 | 每人台詞字數 / 總字數 |
| 場景切換時間 | < 3 秒 | 倒數最後一句台詞 → 舞蹈開始 |

### 定性指標

| 指標 | 目標 | 檢查點 |
|------|------|--------|
| 節奏感 | 無卡頓、加速感明顯 | 讀一遍，描述感受 |
| 笑點保留度 | ≥ 80% 原有笑點 | 列出笑點清單，對勾 |
| 敘事完整度 | 有開始、中間、結尾 | 故事邏輯能說成一段話 |
| 小演員友善度 | 背台詞 ≤ 3 次過 | 排練記錄 |
| 舞蹈銜接自然度 | 無突兀 | 觀眾提問：轉場卡不卡？ |

## 常見陷阱（Pitfall Prevention）

| 陷阱 | 徵兆 | 修復 |
|------|------|------|
| **砍字過度** | 感覺「2 倍速」、「喘不過氣」 | 加入「反應時間」：掌聲、短停頓、眼神對接 |
| **笑點模糊** | 觀眾沒反應 | 保留笑點的「觸發詞」，寧可減其他地方 |
| **舞蹈銜接突兀** | 感覺「說完就跳」 | 用「決定性台詞」做觸發：「那就比啦！」 |
| **多人對白混亂** | 難以分辨誰在說 | 每人每句前加「名字：」，小孩子好區分 |
| **時間超超超** | 現場時間不夠 | 檢查舞蹈長度有沒有偷長；檢查台詞有沒有人加詞 |

## 範本應用

### 新劇本簡化請求時：

```
1. 複製本協議
2. 填入「輸入」欄（時長、人數、舞蹈段落）
3. 依 Phase 1-5 逐步執行
4. 在「品質指標」表中打勾
5. 生成精簡版
6. 回溯性測試（排練）
```

### 反覆迭代時：

```
迭代循環：
  1. 收集排練反饋（卡詞？節奏快？笑點出現？）
  2. 定位問題（見「常見陷阱」表）
  3. 微調（通常不超過 5 句話修改）
  4. 再排練測試
  直到品質指標全部勾選 ✅
```

## 未來 Skill 化（SkillForge Blueprint）

**Skill 名稱**：ScreenplayForge / 劇本煉金術

**輸入** (io):
- 原劇本（markdown / plaintext）
- 目標時長（秒）
- 演出角色列表
- 必保留笑點清單

**過程** (connects_to):
1. DSE：第一性原理分析劇本結構
2. 節奏設計：生成節奏曲線圖
3. 台詞精簡：逐場景壓縮
4. 驗證：品質檢查清單

**輸出** (memory):
- 簡化版劇本（markdown）
- 節奏設計文檔（含圖）
- 品質檢查報告
- 排練提示卡片

**Hub 歸位**：`skillforge`（內容生成）

**預期使用者**：
- 老師 / 舞蹈教室（臨時時間調整）
- 家長（孩子學校表演改期）
- 企業活動策劃（演出縮編）
