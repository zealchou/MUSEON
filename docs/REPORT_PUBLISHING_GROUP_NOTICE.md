# 報告發布流程 — 群組通知（2026-04-01）

## 從今天起：只走原生 GitHub，不走 CDN 中轉

過去一小時的多次驗證中發現，Fastly CDN 快取陷阱導致「我這邊通了、台灣端 404」的現象重複發生。決定從此刻起，**報告發布流程簡化為單一路徑**：

### ✅ 標準做法（從現在起）

```
報告完成
  ↓
python ~/MUSEON/scripts/publish-report.py
  ↓
等待腳本驗證 HTTP 200（自動完成，2 分鐘內）
  ↓
腳本輸出 VERIFIED_URL=https://...github.io/...
  ↓
複製這個連結，直接傳給客戶/群組
  ↓
完成
```

### ❌ 不要做的事

- 不要手動傳 htmlpreview.github.io/jsDelivr 中轉連結
- 不要 curl 自己驗證（腳本已包含）
- 發完就急著跟客戶說「連結已寄」，而是**等 1-2 分鐘確保 GitHub CDN 同步**

### 如果連結無法訪問

1. **第一步**：在台灣網路環境（手機 4G、VPN 出台灣）重新 curl 驗證
2. **第二步**：等 2-3 分鐘，GitHub 自動同步
3. **第三步**：若還是 404，先不傳連結，找 Zeal 確認

### 為什麼這樣做

| 問題 | 舊方式 | 新方式 |
|------|-------|-------|
| 快取延遲 | 5-10 分鐘不確定何時生效 | 腳本自動驗證確認 |
| 多層中轉 | Fastly → jsDelivr → 使用者 | GitHub → 使用者（單層） |
| 故障排查 | 不知道卡在哪一層 | 唯一責任人是 GitHub |
| Telegram 相容 | 有些 URL 格式會被截斷 | 原生 GitHub 格式最穩定 |
