系統範圍定義

MUC MVP 第一階段的範圍是「收費閉環 + 設備授權閉環 + 最低防破解基線」，精準鎖定 12 個核心系統功能。

✅ In Scope（必須做）

  • 帳號與身分驗證 — 註冊、登入、狀態管理
  • Billing Profile — 付款資料建立與維護
  • License / Serial 管理 — 序號建立、查詢、啟用/停用
  • 方案管理（Plan） — 訂閱/續約方案、優惠機制
  • Subscription 週期 — ACTIVE / GRACE / SUSPENDED 狀態
  • 人工收款與續期 — payment_status (DRAFT/CONFIRMED/VOID)
  • Device Pairing — 一次性配對碼、設備綁定、token 發行
  • Device Token / Entitlement — Hash 儲存、可驗章授權
  • Pre-Job Check — 接新任務前的授權再驗證
  • Device Revocation — 設備停用、換機流程
  • 告警中心與通知 — 收款、到期、風控事件
  • 稽核紀錄 — Append-only 日誌

✕ Out of Scope（先不做）

  • 自動金流串接
  • 自動退款
  • 推薦分潤
  • 介紹人折抵
  • 經銷商分潤
  • 國家代理 / 區域代理
  • 客戶資料上傳母體
  • 直接讀取客戶資料庫
  • 複雜報表
  • AI 對話儀表板
  • 自動更新發佈平台
  • 多國稅制 / 國際金流
  • 複雜企業組織與多層審批
  • 深度反逆向 / 反 hook
  • 完整企業級 IAM / SSO
  • 多租戶代理商管理

核心使用情境（8 大流程）

情境 1：新用戶開通

使用者註冊 → 建立付款資料 → 平台建立序號 → 確認收款 → Subscription 生效

情境 2：設備首次配對

使用者生成配對碼 → Agent 輸入配對碼 → 平台完成綁定 → 回傳 device token

情境 3：正常授權輪詢

Agent 呼叫 entitlement API → 平台驗證 token / device_id / fingerprint → 回傳可驗章的授權

情境 4：接新任務前檢查

Agent 呼叫 pre-job check → 平台再次驗證 → 若授權有效則允許接新任務

情境 5：續費

使用者收到提醒 → 客戶付款 → 平台人工確認 → 系統更新期限

情境 6：到期與停用

Subscription 到期 → GRACE 寬限 → SUSPENDED 停用 → Agent 停止接新任務

情境 7：設備停用 / 換機

平台/使用者停用舊設備 → 舊 token 失效 → 新設備重新配對

情境 8：異常與風控

偵測到 fingerprint mismatch / nonce replay / 簽章驗證失敗 → 系統建立告警 → 依風險等級採取行動

最低安全基線（8 項）

  • 序號不能直接當授權憑證 — 必須透過 token / pair code
  • Pair code：一次性、短時效、只存 hash
  • Device token：高熵亂數、只存 hash、可 revoke
  • Token 綁 device_id 與 fingerprint 檢查
  • Entitlement response 必須簽章 — Ed25519
  • Agent 端必須內建公鑰驗章 — 離線驗證
  • 每次接新任務前必查授權或 grant
  • 離線超過 15 分鐘後停止接新任務

外部整合邊界

Agent / Device API

  • POST /api/device/pair — 配對與註冊設備
  • GET /api/device/entitlement — 取得短效可驗章授權
  • POST /api/device/check-before-job — 接新任務前確認授權

Admin API / 後台操作

  • 建立與管理序號
  • 建立與確認收款
  • 查看與停用設備
  • 查看告警與稽核紀錄

排程 / Reconciler

  • ACTIVE → GRACE 狀態校正
  • GRACE → SUSPENDED 狀態校正
  • 通知觸發 + 風控清理

Notification / Invoice Worker

  • 收款確認後通知
  • 到期 / 寬限 / 停用通知
  • 發票後續任務掛點（第一階段保留介面，暫不串接）

金流邊界（第一階段）

  • ❌ 不串接外部 payment gateway
  • ✅ 收款來源：人工匯款或人工對帳
  • ✅ 系統只支援人工確認與狀態更新

安全邊界

  • Agent 不可信任任意可編輯的 license server URL
  • TLS 必要但非唯一信任來源
  • 平台提供可驗章 response
  • Agent 內建平台公鑰並在本地驗章
  • 暫時不納入深度 anti-debug / anti-hook，但保留後續擴充空間

核心結論

MUC MVP 第一階段應嚴格鎖定在：

帳號 + 序號 + Subscription + Device Pairing + Manual Payment + Entitlement + Pre-Job Check + Alerts + Audit Log

只要先把上述範圍做穩，就足以支撐第一階段上線、開始收費、開始控管設備合法使用權。
