系統範圍定義（SYSTEM_SCOPE）

明確定義 MUC MVP 第一階段的包含範圍與排除範圍，以及主要使用情境。

包含範圍（In Scope）

1. 帳號與身分驗證
使用者註冊、登入/登出、基本帳號狀態管理、登入前台查看授權與設備資訊
2. Billing Profile / 付款資料
使用者建立與維護付款資料，平台可查閱以進行人工對帳與開通
3. License / Serial 管理
平台建立序號（可指定方案），使用者查詢自己持有的序號，平台查看序號綁定設備，平台可啟用/停用/作廢序號
3.5 方案管理（Plan）
訂閱方案：A(設定+2月) / B(設定+6月) / C(設定+12月) / D(客製)
續約方案：1月 / 6月 / 12月
每個方案可設定一種優惠（送月/送點/折扣，三選一不可疊加）
信任點：300 點/月，超額 USD 0.1/點
4. Subscription 週期與狀態管理
獨立訂閱週期、狀態支援 ACTIVE / GRACE / SUSPENDED
5. 人工收款與續期
建立人工收款紀錄，支援 payment_status：DRAFT / CONFIRMED / VOID
6. Device Pairing / 設備綁定
一次性配對碼、平台建立設備記錄、成功配對回傳一次性 device token
7. Device Token / Entitlement
Token 只存 hash、可被 revoke、Entitlement 必須回傳可驗章之授權結果
8. Pre-Job Check / 接單前檢查
Agent 每次接新任務前必須再次檢查授權
9. Device Revocation / 換機與停用
支援 device_status：ACTIVE / REVOKED / REPLACED
10. 告警中心與通知
收款成功/失敗事件、到期/寬限/停用事件、設備異常事件、entitlement 驗證失敗事件、風控事件（fingerprint mismatch、nonce replay、token 異常）
11. 稽核紀錄
Append-only 日誌，涵蓋收款確認、設備停用、重綁設備、修改到期日、付款資料變更、pair code 產生/使用/失敗、token 發行/revoke/rotation、entitlement 驗證失敗
12. 最低安全基線
• 序號不能直接當授權憑證
• pair code 一次性、短時效、只存 hash
• device token 高熵亂數、只存 hash、可 revoke
• token 綁 device_id 與 fingerprint 檢查
• entitlement response 必須簽章
• Agent 端必須內建公鑰驗章
• 每次接新任務前必查授權或 grant
• 離線超過 15 分鐘後停止接新任務

主要使用情境（8 個 BDD 場景）

  1. 情境 1：新用戶開通 — 用戶註冊 → 付款資料 → 序號 → 收款確認 → Subscription 生效
  2. 情境 2：設備首次配對 — 產生配對碼 → Agent 輸入 → 設備綁定 → Token 回傳
  3. 情境 3：正常授權輪詢 — Agent 呼叫 entitlement API → 驗證 → 回傳可驗章結果
  4. 情境 4：接新任務前檢查 — Pre-job check → 驗證設備 → 允許/拒絕接單
  5. 情境 5：續費 — 用戶付款 → 平台確認 → 更新期限 → Agent 取得新狀態
  6. 情境 6：到期與停用 — 到期 → GRACE → SUSPENDED → 停止接新任務
  7. 情境 7：設備停用 / 換機 — 停用舊設備 → Token 失效 → 新設備重新配對
  8. 情境 8：異常與風控 — Fingerprint mismatch / Nonce replay → 建立告警 → 採取措施

整合邊界（外部系統 / API）

1. Agent / Device API

  • POST /api/device/pair — 配對與註冊設備
  • GET /api/device/entitlement — 取得短效且可驗章的授權資訊
  • POST /api/device/check-before-job — 在接新任務前再次確認授權狀態

2. Admin API / 後台操作 API

  • 建立與管理序號
  • 建立與確認收款
  • 查看與停用設備
  • 查看告警與稽核紀錄
  • POST /admin/devices/{id}/revoke

3. 排程 / Reconciler

  • ACTIVE → GRACE 狀態校正
  • GRACE → SUSPENDED 狀態校正
  • 通知觸發
  • 例行風控與清理任務

4. 通知 / Invoice Worker

  • 收款確認後通知
  • 到期 / 寬限 / 停用通知
  • 發票後續任務掛點

5. 金流邊界

第一階段不串接外部 payment gateway，收款來源為人工匯款或人工對帳，系統只需支援人工確認與狀態更新。

6. 安全邊界

Agent 不可信任任意可編輯 license server URL，TLS 為必要但非唯一信任來源，平台需提供可驗章 response，Agent 需內建平台公鑰並在本地驗章。
