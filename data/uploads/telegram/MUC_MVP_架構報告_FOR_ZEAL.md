# MUC MVP 第一階段——架構說明書
**專案名：** MUC (Multi-User/Multi-Client?)
**負責人：** Feng
**階段目標：** 上線收費 + 設備授權管理 + 基礎防破解
**發布日期：** 2026-03-23

---

## 一句話版本

**Feng 在建一套「序號授權系統」**：使用者買訂閱→平台給序號→設備綁序號→設備定期驗證授權→到期停用。

---

## 系統四個角色

| 角色 | 做什麼 | 誰負責 |
|------|--------|--------|
| **使用者** | 註冊帳號、付款、看授權狀態 | 前台 UI |
| **設備（Agent）** | 定期問「我還有授權嗎？」 | 手機 App / 硬體 |
| **平台後端** | 驗證授權、管理序號、處理收款 | Feng 的 API |
| **管理員（你們）** | 確認收款、停用設備、看告警 | 後台儀表板 |

---

## 11 個核心功能模塊

### 第一層：帳號體系
1. **帳號註冊/登入** — 標準 auth
2. **付款資料管理** — 存客戶銀行帳號（人工對帳用）

### 第二層：序號與訂閱
3. **序號（License）** — 產品金鑰。一個序號 = 一個客戶一台設備的授權
4. **訂閱方案（Plans）**
   - **新客戶選這些：**
     - A方案：設定費 NT$24,000 + 2個月用度
     - B方案：設定費 NT$24,000 + 6個月用度
     - C方案：設定費 NT$24,000 + 12個月用度
     - D方案：客製化（金額和期限你們決定）
   - **既有客戶選這些（續費）：**
     - 純月費 NT$6,000/月（1個月、6個月、12個月選項）
   - **所有方案都含：** 每月 300 信任點（超額的話 USD 0.1/點）
   - **優惠規則：** 送月 或 送點 或 打折（三選一，不能疊加）

5. **訂閱週期管理** — 每個序號獨立計費期。到期→寬限→停用

### 第三層：設備控制
6. **設備配對** — 使用者手機掃碼→平台給 device token（一次性，明文只回傳一次）
7. **Device Token** — 設備身份證。只存 hash，可被撤銷
8. **接單前檢查** — 設備每次接新工作前都要問一次「我還有授權嗎？」

### 第四層：安全防禦
9. **設備停用/換機** — 舊 token 立即失效，新設備重新配對
10. **告警系統** — 收款/到期/異常事件，管理員可確認
11. **稽核紀錄** — 記錄所有關鍵動作（誰停用了設備、何時重新綁定、簽章驗證失敗等）

## 業務流程：8 個真實場景

### 🆕 場景 1：新客戶開通（5 步）
```
使用者註冊
    ↓
上傳付款資料（銀行帳號）
    ↓
你們看到→人工確認收款
    ↓
系統自動建立序號
    ↓
使用者可在前台看到「已啟用」狀態
```

### 📱 場景 2：設備首次連接（6 步）
```
使用者在手機點「新增設備」
    ↓
系統產生「一次性配對碼」（15 分鐘有效）
    ↓
用戶在手機 App 輸入配對碼
    ↓
平台驗證成功 → 綁定設備
    ↓
平台回傳 device token（明文只傳一次！）
    ↓
手機 App 定期用這個 token 問「我還有授權嗎？」
```

### ✅ 場景 3：設備正常授權輪詢（每次都做）
```
設備定期發送：device token + device ID + 指紋 + 時間戳
    ↓
平台驗證（簽章有沒有被篡改）
    ↓
平台回傳：「yes，你還有授權」（帶簽章，設備驗證簽章）
    ↓
設備更新本地授權狀態
```

### 🔄 場景 4：接新任務前再檢查
```
設備要接新工作
    ↓
先問一次「我的授權還有效嗎？」
    ↓
平台說「有效」或「無效」
    ↓
有效 → 接受新任務 / 無效 → 拒絕接新任務
```

### 💰 場景 5：客戶續費
```
到期前 → 發通知給客戶
    ↓
客戶匯款
    ↓
你們看到匯款 → 在後台點「確認收款」
    ↓
系統自動延長訂閱期限
    ↓
設備下次輪詢就能看到新狀態
```

### ⏰ 場景 6：過期與停用
```
訂閱到期
    ↓
系統標記為「GRACE」（寬限期 7 天左右）
    ↓
設備還能完成已接的任務，但不能接新任務
    ↓
寬限期過了 → 標記為「SUSPENDED」
    ↓
設備完全停用
```

### 🚨 場景 7：設備停用或換機
```
【停用舊設備】
使用者點「停用設備」
    ↓
該 device token 立即失效
    ↓
設備下次輪詢 → 授權查詢失敗 → 停止接新任務

【換新設備】
新設備必須重新走「配對碼」流程
    ↓
只有配對成功才能取得新 token
    ↓
不能直接複製舊 token 給新設備
```

### 🚨 場景 8：異常與風控
```
偵測到異常行為（如簽章驗證失敗、設備指紋不符、token 被盜用）
    ↓
系統建立「告警」
    ↓
根據風險等級：
  - 低級 → 只告警
  - 中級 → 要求使用者重新配對
  - 高級 → 直接撤銷 token
```

## 安全底線：12 個防護要求

Feng 已內建的防破解機制（第一階段最低要求）：

| # | 防護要求 | 為什麼 |
|---|---------|--------|
| 1 | 序號≠授權憑證 | 不能直接拿序號去驗證，要經過 device token 流程 |
| 2 | 配對碼一次性 + 短時效 | 防止有人截獲碼再拿去配對其他設備 |
| 3 | Device token 是高熵亂數 | 很難被猜中 |
| 4 | Token 只存 hash | 即使資料庫洩露，駭客也看不到明文 token |
| 5 | Token 綁 device_id + 指紋 | 設備不一致就驗證失敗 |
| 6 | 授權 response 必須簽章 | 防止平台 API 被中間人攻擊 |
| 7 | 設備內建公鑰驗章 | 設備自己驗，不靠 SSL 憑證 |
| 8 | 每次接新任務再檢查授權 | 防止使用者邊接任務邊改設備狀態 |
| 9 | 離線超 15 分鐘停止接任務 | 防止設備離線時繼續執行（已被停用的任務） |
| 10 | 告警系統自動偵測異常 | fingerprint 不符、token 重放、簽章失敗都會被抓 |
| 11 | 稽核日誌（append-only） | 記錄每一筆關鍵動作，不可刪除 |
| 12 | 不提供直接刪除 audit log 的 API | 防止有人試圖銷毀證據 |

## 關鍵決策：做什麼 & 不做什麼

### ✅ MVP 第一階段「一定要做」

- ✅ 帳號系統（註冊、登入、付款資料）
- ✅ 序號與訂閱週期管理
- ✅ 設備配對與 token 發行
- ✅ 授權輪詢 + 簽章驗證
- ✅ 接新任務前檢查
- ✅ 設備停用/換機流程
- ✅ 告警與稽核紀錄
- ✅ 人工收款確認流程（不自動扣款）
- ✅ 後台管理界面（確認收款、停用設備、查告警）

### ❌ 第一階段「明確不做」

- ❌ 自動金流串接（PChome、綠界之類）
- ❌ 自動退款
- ❌ 經銷商分潤、代理商模式
- ❌ 複雜的企業組織與多層審批
- ❌ AI 對話儀表板（留給第二階段）
- ❌ 深度反逆向工程（anti-hook / anti-debug）

**決策理由：** 第一階段的目標是「能上線、能收費、能控管」。不要一次做完，先把核心路徑跑順。

## Feng 需要的 API 邊界

### 設備端需要的 API（3 個）
```
POST /api/device/pair
  ← 裝置傳配對碼
  → 平台回傳 device token

GET /api/device/entitlement
  ← 裝置傳 token + device_id + 指紋 + nonce + 時間戳
  → 平台回傳簽章後的授權結果

POST /api/device/check-before-job
  ← 裝置要接新任務前檢查
  → 平台回傳「可接」或「不可接」
```

### 你們後台需要的 API（最少 7 個）
```
建立序號、停用序號
建立付款紀錄、確認收款
查看設備清單、停用設備
查看告警列表、標記已讀
查看稽核紀錄
```

### 後台排程任務（自動運行）
```
ACTIVE → GRACE 狀態校正（看到期日就轉）
GRACE → SUSPENDED 狀態校正（寬限期過了就轉）
發通知：「快到期了」/ 「已過期」/ 「已停用」
風控檢查：偵測異常並建告警
```

## 下一個疑問（先搶答）

**Q: 為什麼一個序號只能一台 ACTIVE 設備？**
A: 防止使用者用一個帳號開 10 台設備。業務邏輯上，一個 license = 一台裝置的授權權。想多設備的話，客戶要多買序號。

**Q: Device token 為什麼發行一次就算，設備要存好？**
A: 防止網路攔截。token 明文只傳一次，之後設備存在本地，每次輪詢時再送。

**Q: 簽章驗證放在設備端會不會被破解？**
A: 會。但第一階段就是這樣。要完全防逆向需要硬體安全晶片，那是第二階段的事。

**Q: 人工確認收款會不會很麻煩？**
A: 短期會。等客戶數穩定後，再接自動金流 gateway（PChome、綠界）。
