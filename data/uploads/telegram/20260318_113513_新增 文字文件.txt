  ---
  各方案的甜頭與代價

  A 現狀
  - 甜頭：最安全，不用改任何東西
  - 代價：使用者操作最多，新手容易迷路（「配對碼在哪裡？」）

  B 成功頁顯碼
  - 甜頭：改動最小（只改一個前端頁面），流程順暢
  - 代價：如果使用者付完錢還沒裝 Agent，碼就白建了；仍需手動複製貼上

  C Agent 自動配對（現有 API）
  - 甜頭：後端零改動，使用者體驗最好
  - 代價：Agent 要串 4 支 API（登入 → 查 License → 建碼 → 配對），任一步失敗要處理；Agent 短暫持有帳密

  D Agent 自動配對（專用端點）
  - 甜頭：Agent 最簡單（只要登入 + 一個呼叫），錯誤處理集中在後端
  - 代價：要新增後端 API，等於多一個要維護的端點

  ---
  建議

  先做 B，再做 C，D 暫時不做。

  理由：

  B（成功頁顯碼）
    ├─ 改動量：最小（前端 1 個檔案）
    ├─ 立即改善體驗：使用者付完錢馬上看到下一步
    └─ 30 分鐘內可完成

  C（Agent 自動配對，用現有 API）
    ├─ 後端零改動，只需要寫 Agent 端程式碼
    ├─ 對「進階使用者」或「批量部署」場景很有用
    └─ 跟 B 不衝突，兩條路並存

  D（專用端點）
    └─ 等 C 驗證流程跑通、確認 4 步 round trip 是瓶頸時再做
        MVP 階段不需要過早優化

  B + C 並存的好處：使用者有兩種配對方式可選——
  - 習慣 GUI 的人：成功頁複製碼貼到 Agent
  - 習慣 CLI 的人：Agent 輸入帳密自動搞定
