## 詳細說明

### Phase 1：基礎建設（順序 1–3）✅

#### 1. 專案骨架
- **產出**：`backend/`（NestJS Modular Monolith）+ `frontend/`（Vue 3 + Vite + Tailwind）
- **已完成**：11 個模組骨架、路由分層（Portal / Admin）、設定檔

#### 2. Migration：accounts
- **產出**：`src/database/migrations/1741580000000-CreateAccounts.ts`
- **欄位**：id / email / passwordHash / name / status / role / createdAt / updatedAt

#### 3. Auth 模組
- **產出**：
  - `POST /api/v1/auth/login`（回傳 JWT）
  - `POST /api/v1/auth/register`
  - `GET /api/v1/auth/me`（JWT Guard 驗證）
  - `JwtAuthGuard` / `AdminGuard` / `@CurrentUser()` 裝飾器

---

### Phase 2：License & Subscription（順序 4–9）

#### 4. Migration：licenses + pair_codes
- **產出**：兩張表
- **關鍵欄位**：
  - licenses：id / serialKey / accountId / status
  - pair_codes：id / licenseId / codeHash / expiresAt / usedAt

#### 5. License 模組（Admin）
- **產出**：
  - `POST /api/v1/licenses/admin`（平台建立序號）
  - `GET /api/v1/licenses/my`（User 查詢自己的序號）

#### 6. Pair Code 生成
- **產出**：`POST /api/v1/licenses/:id/pair-code`
- **規則**：一次性、只存 hash、明文只回傳一次（ref：STATE_RULES T-PAIR-001）
- **你需要確認**：OQ-005（TTL 預設 5 分鐘，是否拍板）

#### 7–8. Migration + Subscription 狀態機
- **產出**：subscriptions 表 + ACTIVE/GRACE/SUSPENDED 轉換邏輯
- **依據**：STATE_RULES §3

#### 9. Subscription Reconciler
- **產出**：`@Cron` 排程，每 5 分鐘檢查
- **規則**：冪等、不重複觸發 side effects（ref：STATE_RULES §9）

---

### Phase 3：Payment（順序 10–12）

#### 10–11. Migration + Payment 模組
- **產出**：payments 表 + DRAFT/CONFIRMED/VOID 狀態 API
- **你需要確認**：OQ-004（Payment confirmation 是否強制冪等）

#### 12. Payment → Subscription 連動
- **產出**：`adminConfirm()` 成功後觸發 `subscriptionService.activateOrRenew()`
- **規則**：CONFIRMED 不可再 void；不得重複延長週期

---

### Phase 4：BillingProfile（順序 13–14）

- **產出**：billing_profiles 表 + CRUD API
- **用途**：User 維護發票資料（公司名稱、統編、地址）

---

### Phase 5：Device Pairing（順序 15–17）

#### 16. Device Pairing（核心）
- **產出**：
  - `POST /api/v1/devices/agent/pair`
  - 驗證 PairCode（未過期、未使用、license 無現有 ACTIVE Device）
  - 建立 Device（ACTIVE）+ DeviceToken（只回傳一次明文，存 hash）
- **依據**：STATE_RULES T-DEV-001, T-PAIR-002

#### 17. Device Revoke
- **產出**：
  - `POST /api/v1/devices/:id/revoke`（User 自助）
  - `POST /api/v1/admin/devices/:id/revoke`（Admin）
- **你需要確認**：OQ-008（User 能否自行 revoke）

---

### Phase 6：Audit & Alert（順序 18–20）

- **目標**：所有 STATE_RULES 定義的狀態變更，都必須寫 audit_log
- **Alert 事件映射**（ref STATE_RULES §8.4）：
  - SUBSCRIPTION_GRACE → WARN
  - SUBSCRIPTION_SUSPENDED → CRITICAL
  - NONCE_REPLAY → CRITICAL
  - SIGNATURE_VERIFICATION_FAILED → CRITICAL
  - SAME_TOKEN_MULTI_DEVICE → CRITICAL
- **你需要確認**：OQ-007（分級規則是否拍板）

---

### Phase 7：Entitlement & Security（順序 21–22）

#### 21. Entitlement 模組
- **產出**：
  - `POST /api/v1/agent/entitlement`
  - 回應以 Ed25519 私鑰簽章，Agent 用內建公鑰驗章
  - 回應含 expiresAt（ref：OQ-006，預設 5 分鐘）
- **你需要確認**：OQ-006（TTL 是否拍板）

#### 22. Pre-Job Check
- **產出**：`POST /api/v1/agent/check-before-job`
- **判定邏輯**：STATE_RULES §10（allow_new_jobs 一票否決條件）
- **離線限制**：超過 15 分鐘 → `PRE_JOB_CHECK_FAILED`
