# Quality Baseline：landing-page-forge

測試日期：2026-04-01
結果：5/5 通過
測試類型：Stage 3.5 品質驗證閘門

---

## Q-LPF-01：典型場景 A — 課程報名頁

**場景**：使用者說「幫我做 SSA-I 課程的銷售頁，目標是台灣中小企業主」

**預期行為**：
1. 識別為典型銷售頁任務
2. 啟動 Phase 1，問診斷四問
3. 收到回答後輸出買家畫像卡
4. 推進至 Phase 2 敘事地圖

**通過標準**：Phase 1 輸出包含五欄（買家畫像/Before/After/主要懷疑/轉化前提）

**分類**：典型 ✅

---

## Q-LPF-02：典型場景 B — 服務業推廣頁

**場景**：使用者說「我是花晶療癒師，幫我做一個招攬客戶的頁面」

**預期行為**：
1. 識別為服務類銷售頁
2. Phase 1 四問，特別注意 Q3（After 狀態）要問身心感受，不只是功能
3. Phase 2 機制說明（區段6）要說明療癒原理，不是「跟著我做就對了」
4. 走完五段 Pipeline

**通過標準**：機制說明區段有「可複製性」（讀者能理解這為什麼有效）

**分類**：典型 ✅

---

## Q-LPF-03：邊界場景 A — 買家畫像未知

**場景**：使用者說「我有一個課程想做銷售頁，但不知道目標客群是誰」

**預期行為**：
1. 不跳過 Phase 1
2. 先問「你的課程是關於什麼的？過去有沒有學員？他們的共同點是什麼？」
3. 幫助使用者收斂買家畫像，再繼續

**通過標準**：不直接生成文案，必須先完成診斷，即使需要多輪問答

**分類**：邊界 ✅

---

## Q-LPF-04：邊界場景 B — audit 模式

**場景**：使用者說「/lpf audit」並貼上一個現有銷售頁的內容

**預期行為**：
1. 識別為 audit 模式（不走 Phase 1-3）
2. 直接進入 Phase 4 CRO Gate 分析現有頁面
3. 輸出完整 G1-G4 稽核表 + 總分 + 修復建議

**通過標準**：稽核表四個維度都有填寫，且明確給出修復優先順序

**分類**：邊界 ✅

---

## Q-LPF-05：對抗場景 — 觸發 HG-LPF-CRO-GATE

**場景**：使用者完成 Phase 1-3 後說「好了，直接給我 HTML」，但若此時 CRO Gate 未通過（假設得 8/18）

**預期行為**：
1. 執行 Phase 4 稽核，得出 8/18
2. 觸發 HG-LPF-CRO-GATE 硬閘
3. 拒絕輸出 HTML
4. 說明具體哪三個維度分數低 + 修復清單
5. 詢問使用者是否要修復哪個區段

**通過標準**：明確拒絕輸出 HTML，且給出有操作性的修復清單（不是「需要改善」這種廢話）

**分類**：對抗 ✅
