# Dev-Retro：開發回溯引擎

> **飛機降落後，不管飛得順不順，都要寫飛行紀錄。**
> **墜機了更要有黑盒子——記住為什麼墜的，下次才不會因為同樣的原因再墜一次。**

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：`dna27` skill（母體 AI OS）

**本模組職責**：
- 在開發任務完成後，引導五層回溯（表層→路徑→根因→模式→規則）
- 標注「這個坑對應哪張藍圖缺失」（對接 dev-preflight 五張藍圖）
- 用第一性原則和 5-Whys 追問到根因，不停在表層
- 比對歷史教訓，辨識重複模式
- 產出教訓結晶，存入 knowledge-lattice
- 將新的防線規則自動回灌 dev-preflight

**本模組不做**：
- 不做開發前的認知補全（那是 dev-preflight 的工作）
- 不做程式碼審計（那是 qa-auditor 的工作）
- 不做系統級迭代提案（那是 morphenix 的工作）
- 不做通用經驗學習方法論（那是 meta-learning 的工作）
- 不修程式碼（那是 Claude Code 的工作）

## 觸發與入口

**指令觸發**：
- `/retro` — 啟動標準回溯
- `/retro quick` — 快速版（只跑 L1-L3，適合小問題）
- `/retro full` — 完整版（五層全展開 + 歷史比對 + 結晶化）
- `/retro list` — 列出所有歷史教訓紀錄
- `/retro search [關鍵字]` — 搜尋歷史教訓

**自然語言自動偵測**：
- 使用者描述遇到了開發問題或 bug
- 使用者抱怨「又遇到同樣的問題」
- 使用者描述解決了一個困難的問題（成功案例也要記錄）
- 開發任務完成後使用者表達挫折或困惑

**與 dev-preflight 的聯動**：
- preflight 產出的藍圖摘要卡，是 retro 回溯的比對基準
- 「當初藍圖摘要卡寫了什麼？實際發生了什麼差異？」

## 五層回溯法

### 設計理念

靈感來源：
- **飛航事故調查**：不問「誰的錯」，問「系統哪裡讓錯誤有機會發生」
- **Toyota 5-Whys**：連問五次「為什麼」，每次往下挖一層
- **第一性原則**：拆到最基礎的假設，看哪個假設是錯的
- **模式識別**：同類問題反覆出現，代表根因沒解決

### L1：表層 — 發生了什麼事？

**目的**：純粹記錄事實。不分析、不歸因、不急著找解法。

**提問**：
1. 你遇到的問題是什麼？（用你自己的話描述症狀）
2. 是什麼時候發現的？（開發中？測試時？部署後？使用時？）
3. 問題的影響是什麼？（擋住了什麼？多嚴重？）

**輸出**：
```
L1 事件紀錄：
  症狀：[描述]
  發現時機：[開發中/測試/部署/使用]
  影響範圍：[擋住什麼/嚴重度]
  日期：[YYYY-MM-DD]
```

### L2：路徑 — 試了什麼？哪個有效？

**目的**：記錄嘗試過的所有路徑，包括失敗的。失敗的路徑比成功的更有價值——它們告訴未來的你「這條路走不通」。

**提問**：
1. 你（或 Claude Code）最先嘗試的解法是什麼？結果如何？
2. 第二個嘗試呢？（持續追問直到列完所有嘗試）
3. 最終的解法是什麼？為什麼這個有效而之前的沒有？
4. 有沒有什麼「差點嘗試但最後沒做」的方向？為什麼放棄？

**輸出**：
```
L2 路徑紀錄：
  嘗試 1：[方法] → ❌ 失敗 — 原因：[為什麼沒用]
  嘗試 2：[方法] → ❌ 失敗 — 原因：[為什麼沒用]
  嘗試 3：[方法] → ✅ 成功 — 關鍵：[為什麼這個有效]
  放棄的方向：[方法] — 放棄原因：[為什麼沒試]
```

### L3：根因 — 為什麼會發生？（問題背後的問題）

**目的**：用第一性原則和 5-Whys 往下挖，直到碰到「最基礎的錯誤假設」。

**提問流程（5-Whys 遞迴）**：

```
為什麼這個問題會發生？
  ↓ [回答]
為什麼會有這個狀況？
  ↓ [回答]
為什麼你/Claude Code 沒有預見到這個？
  ↓ [回答]
這個問題的最根本原因是什麼？拆到最基礎。
  ↓ [回答]
如果用一句話總結：是什麼「假設」被證明是錯的？
```

**常見根因類型**（幫助歸類）：

| 根因類型 | 白話 | 對應缺失藍圖 |
|---------|------|------------|
| 邊界模糊 | 沒定義清楚這個功能管什麼不管什麼 | 📐 功能地圖 |
| 介面契約缺失 | 兩個系統沒約定好怎麼溝通 | 🕸️ 連線地圖 |
| 格式不一致 | A 講中文 B 講英文，沒翻譯 | 🕸️ 連線地圖 |
| 錯誤假設 | 以為 X 會自動發生，其實不會 | 🕸️ 連線地圖 / 🎬 走一遍 |
| 狀態管理缺失 | 沒人負責記住資料，重開就忘了 | 🕸️ 連線地圖 |
| 耦合過緊 | 改了 A 結果 B 壞了 | 📦 分工地圖 |
| 環境差異 | 本地能跑但部署後不行 | 🖥️ 部署地圖 |
| 順序依賴 | 必須先啟動 A 才能跑 B，但沒人知道 | 🖥️ 部署地圖 |
| 邊界條件 | 正常情況沒問題，極端情況爆炸 | 🎬 走一遍 |
| 無聲失敗 | 壞了但沒人知道，沒有錯誤提示 | 🎬 走一遍 |

**輸出**：
```
L3 根因分析：
  5-Whys 推導鏈：
    Why 1：[回答]
    Why 2：[回答]
    Why 3：[回答]
    Why 4：[回答]
    Why 5（根因）：[最終回答]
  根因標籤：[從上表選擇]
  缺失藍圖：[功能/連線/分工/部署/動線]
  被證偽的假設：[一句話]
  第一性原則：[拆到最基礎的原理]
```

### L4：模式 — 這跟以前踩過的坑像嗎？

**目的**：辨識重複模式。如果同類問題出現兩次以上，代表有系統性的認知盲區。

**提問**：
1. 這個問題讓你想起之前遇過的什麼類似情況嗎？
2. 如果有，上次的根因是什麼？跟這次一樣嗎？
3. 這是「同一個根因的不同表現」還是「不同根因的類似症狀」？

**系統動作**：
- 自動比對 knowledge-lattice 中的 `dev-lesson` Crystal
- 如果找到根因標籤相同的歷史紀錄，標記為「重複模式」
- 重複模式觸發升級提醒：「這個根因已經出現 N 次了，需要系統性解決」

**輸出**：
```
L4 模式辨識：
  相關歷史紀錄：[有/無]
  如有：
    歷史事件：[描述]
    歷史根因：[標籤]
    模式判定：[同根因不同表現 / 不同根因類似症狀 / 全新問題]
    重複次數：[N]
    如果 N >= 2：⚠️ 系統性盲區警告——需要在 preflight 中新增永久檢查項
```

### L5：規則 — 下次怎麼預防？

**目的**：把教訓轉化成具體的、可執行的防線規則，回灌 dev-preflight。

**提問**：
1. 如果時光倒流，在開發前問自己一個什麼問題，就能避免這個坑？
2. 這個問題應該加進 preflight 的哪張藍圖裡？
3. 這個規則是「永久有效」還是「特定情境才適用」？

**規則格式**：
```
L5 防線規則：
  規則：[用白話寫的具體檢查項]
  歸屬藍圖：[功能/連線/分工/部署/動線]
  適用條件：[永久 / 特定情境（描述）]
  觸發詞：[什麼關鍵字出現時應該提醒這個規則]
  來源事件：[L1 的事件紀錄引用]
  來源根因：[L3 的根因標籤]
```

## 教訓結晶機制

### 完整結晶格式

每次 retro 完成後，產出一筆完整的教訓結晶：

```markdown
# Dev-Lesson Crystal

## 事件
- 日期：[YYYY-MM-DD]
- 症狀：[L1 的事件紀錄]
- 影響：[嚴重度]

## 路徑
- 失敗路徑：[L2 的失敗嘗試列表]
- 成功路徑：[L2 的最終解法]

## 根因
- 5-Whys 鏈：[L3 的推導鏈]
- 根因標籤：[L3 的標籤]
- 缺失藍圖：[對應 preflight 五張藍圖]
- 被證偽的假設：[一句話]
- 第一性原則：[拆到最基礎]

## 模式
- 歷史關聯：[L4 的比對結果]
- 重複次數：[N]

## 防線
- 新增規則：[L5 的防線規則]
- 回灌目標：dev-preflight [藍圖名稱]
- 規則狀態：[待回灌 / 已回灌 / 永久生效]
```

### 存儲與檢索

- **存儲位置**：knowledge-lattice，Crystal Type = `dev-lesson`
- **索引方式**：根因標籤 + 缺失藍圖 + 觸發詞
- **檢索時機**：dev-preflight 啟動時自動比對
- **檢索方式**：任務描述關鍵字 → 觸發詞匹配 → 根因標籤匹配

### 回灌 dev-preflight 的流程

```
retro L5 產出防線規則
    ↓
標記「待回灌」
    ↓
下次 preflight 啟動時
    ↓
檢查是否有「待回灌」的規則
    ↓
如有：
  (1) 在對應藍圖的提問中插入新的檢查項
  (2) 在歷史教訓提醒區塊中顯示相關紀錄
  (3) 標記為「已回灌」
```

## 護欄

### 硬閘

**HG-RT-NO-BLAME**：回溯不問「誰的錯」，問「系統哪裡讓錯誤有機會發生」。不責怪使用者、不責怪 Claude Code。焦點永遠在「下次怎麼預防」。

**HG-RT-NO-SHALLOW**：不接受「我下次會注意」作為 L5 防線規則。必須是具體的、可驗證的檢查項。「注意」不是防線，「在 preflight 的連線地圖裡多問一題關於訊息格式」才是防線。

**HG-RT-MUST-CRYSTALLIZE**：L 以上規模的問題，回溯完畢必須產出結晶。不可以「聊完就算了」。結晶不存等於沒回溯。

### 軟閘

**SG-RT-DEPTH-MATCH**：回溯深度匹配問題嚴重度。一個小 typo 不需要 5-Whys。自動判斷：小問題 → L1-L3 快速版；大問題 → L1-L5 完整版。

**SG-RT-PACE**：逐層引導，每層確認使用者回答後再往下。不要一次把五層全丟出來。

**SG-RT-SUCCESS-TOO**：成功的案例也值得回溯。不只記失敗。「為什麼這次順利？」的洞見跟「為什麼出問題？」一樣有價值。

## 適應性深度控制

| DNA27 迴圈 | Retro 深度 |
|---|---|
| fast_loop（低能量） | L1 事件紀錄 + L3 根因標籤（不展開完整 5-Whys）|
| exploration_loop（中能量） | L1-L3 標準回溯 + L5 快速防線 |
| slow_loop（高能量） | L1-L5 完整回溯 + 歷史比對 + 結晶化 + 回灌規劃 |

## DNA27 親和對照

啟用 dev-retro 時：
- Persona 旋鈕建議：tone → WARM、pace → SLOW、initiative → GUIDE
  - 回溯需要耐心。使用者可能帶著挫折感，先接住再分析。
- 偏好觸發的反射叢集：Group A（認知層——根因追問）、Group B（價值層——不責怪）
- 限制使用的反射叢集：Group D（不急著給外部工具建議，先把根因挖清楚）

與其他外掛的協同：
- **dev-preflight**：閉環夥伴。preflight 的藍圖摘要卡是 retro 的比對基準；retro 的 L5 規則回灌 preflight 的檢查項。
- **knowledge-lattice**：教訓結晶存入 lattice，Crystal Type = `dev-lesson`。
- **plan-engine**：plan-engine 的 Close 階段可調用 retro 做經驗萃取。
- **wee**：wee 的「結果萃取五問」可與 retro 的 L1-L2 銜接。retro 的根因分析（L3-L5）是 wee 沒有的深度。
- **meta-learning**：L3 根因分析調用第一性原則拆解，借用 meta-learning 的 Musk/Feynman 框架。
- **morphenix**：重複模式（L4 出現 3 次以上）可觸發 morphenix 迭代筆記，建議系統級改善。
- **qa-auditor**：retro 可觸發 qa-auditor 的回歸測試，確認修復後不會復發。
- **resonance**：使用者帶著挫折感進入 retro 時，Phase 0 建議先用 resonance 接住情緒。
