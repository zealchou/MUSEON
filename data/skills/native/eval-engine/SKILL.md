# Eval-Engine：效能儀表板 — 系統化品質度量引擎

> **沒有度量就沒有真正的演化。**
> 感覺變好了不算，數據說變好了才算。
> morphenix 負責「改」，Eval-Engine 負責「證明改對了」。

---

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：`dna27` skill（母體 AI OS。本模組不可脫離 dna27 獨立運作。）

**深度耦合模組**：
- `deep-think` — Phase 2 四道自我檢查作為即時品質信號來源
- `morphenix` — 迭代前後效能比對的主要消費者；Skill 健康儀表板的數據供應商
- `wee` — 工作流熟練度指標互相校準
- `knowledge-lattice` — 結晶共振指數作為知識應用效果度量
- `orchestrator` — 客戶旅程品質度量；使用率熱力圖為編排優化提供依據

**本模組職責**：
- 定義可量化的系統品質 KPI
- 即時評分每次回答的品質（多維度）
- 追蹤品質趨勢（日/週/月粒度）
- 迭代前後 A/B 比對——用數據驗證改動效果
- 偵測系統性弱項（領域/情境/模式/時序盲點）
- 為 morphenix 提供「該改什麼」的數據依據
- 為使用者提供可讀的品質報告
- 追蹤企業客戶導入後的 KPI 前後測變化
- 度量 Skill 實際使用分布，識別閒置與高頻 Skill

**本模組不做**：
- 不做即時品質控制（那是 deep-think 的工作）
- 不做系統迭代（那是 morphenix 的工作）
- 不獨立決定「好壞」——只提供數據，判斷權歸使用者
- 不做客戶的實際記帳或報稅（那是外部工具的工作）

## 一、品質 KPI 定義

### 1.1 五大品質維度

| 維度 | 說明 | 數據來源 | 計算方式 |
|------|------|---------|---------|
| **Q-Score** | 單次回答綜合品質 | deep-think Phase 2 | 四道審計加權平均（0-1） |
| **Skill 命中率** | Skill 是否正確匹配意圖 | 使用者行為 | 採納數 / 觸發總數 |
| **任務完成率** | 使用者任務是否被解決 | 結束方式 + 滿意信號 | 正向結束 / 總任務數 |
| **滿意度代理** | 隱性滿意度推估 | 對話行為模式 | 複合指標（見 1.2） |
| **Token 效率** | 同等品質下的資源消耗 | Token / Q-Score | 品質密度 = Q-Score / kToken |

### 1.2 滿意度代理指標

直接問「你滿意嗎」最不靠譜。用行為信號推估：

**正向信號**（+1.0）：回答後直接行動、確認性語句、同話題深入、後續引用結論

**負向信號**（-1.5，一次失望影響 > 一次滿意）：重複問同題、否定修正、對話中斷、手動改方向

```
Satisfaction = (正向×1.0 - 負向×1.5) / 總信號數
# > 0.5 健康 | 0~0.5 關注 | < 0 告警
```

### 1.3 Q-Score 計算

基於 deep-think Phase 2 四道自我檢查：

| 檢查項 | 子維度 | 權重 |
|--------|-------|------|
| 真問題檢查 | 問題理解 | 0.30 |
| 自動反應檢查 | 思考深度 | 0.25 |
| 溝通品質檢查 | 表達清晰 | 0.20 |
| 可行動性檢查 | 實用性 | 0.25 |

```
Q-Score = 0.30×理解 + 0.25×深度 + 0.20×清晰 + 0.25×實用
# > 0.7 優秀 | 0.5-0.7 及格 | < 0.5 需改善
```

## 五、子系統四：盲點雷達（Blindspot）

掃描系統性弱項——不是某次不好，而是某個維度持續偏低。

### 5.1 盲點類型

| 類型 | 偵測方式 | 範例 |
|------|---------|------|
| 領域盲點 | 某領域 Q-Score 低於均值 20%+ | 財務議題持續低 |
| 情境盲點 | 某類意圖 Skill 命中率低 | 「比較方案」常誤觸 |
| 模式盲點 | 某子維度持續偏低 | 實用性長期低分 |
| 時序盲點 | 品質隨輪次下降 | 長對話後半段退化 |
| Skill 盲點 | 某 Skill 採納率持續低 | 觸發但被忽略 |

### 5.2 輸出範例

```
🔍 盲點掃描（30天 / 47次對話）
━━━━━━━━━━━━━━━━━━━━

🔴 高優先：
  1. 領域：「財務」Q-Score 0.52（均值 0.76）
     → 建議 knowledge-lattice 補充財務結晶
  2. 模式：「實用性」連續 2 週下降
     → 建議 deep-think 加強可行動性審計

🟡 關注：
  3. 時序：20+ 輪對話 Q-Score 降 14%
     → 建議長對話定期摘要重聚焦

已同步至 morphenix 迭代筆記。
```

## 六、子系統五：客戶成效追蹤器（Phase 2 新增）

### 6.1 問題：MUSEON 對客戶真的有用嗎？

系統品質是「我們回答得好不好」，客戶成效是「客戶的生意真的變好了嗎」。兩者不一樣。政府 AI 輔導計畫申請、行銷案例報告、客戶續約決策——都需要量化的成效數據。

### 6.2 客戶 KPI 框架

根據 business-12 的 12 力框架，定義中小企業通用 KPI 池：

| 維度 | KPI 範例 | 數據來源 |
|------|---------|---------|
| 營收 | 月營收、客單價、回購率 | 客戶自報 |
| 行銷 | 觸及數、互動率、轉換率、CPA | 客戶自報 / 平台數據 |
| 品牌 | 品牌搜尋量、NPS、口碑提及數 | 客戶自報 |
| 營運 | 出貨天數、客訴率、翻桌率、稼動率 | 客戶自報 |
| 人資 | 留任率、招募週期、員工滿意度 | 客戶自報 |
| 財務 | 毛利率、現金流天數、損益比 | 客戶自報 |

### 6.3 前後測流程

```
客戶導入
    ↓
Step 1: 基準線建立（/eval client baseline）
    ├ 引導客戶填入 3-5 個核心 KPI 的「現況值」
    ├ 記錄日期、產業、規模
    └ 產出基準線文件
    ↓
（服務期間：1-3 個月）
    ↓
Step 2: 成效測量（/eval client compare）
    ├ 引導客戶更新同一組 KPI 的「現在值」
    ├ 自動計算變化量和變化率
    └ 產出前後測比對報告
```

### 6.4 成效報告輸出

```
📊 客戶成效報告
━━━━━━━━━━━━━━━━━━━━
客戶：[名稱] | 產業：[產業] | 導入期：[起訖日]
服務內容：品牌定位 + 社群內容策略 + 行銷自動化規格

── KPI 前後測 ──
                      基準線    現在     變化
月營收               $120K    $156K    +30% 🟢
IG 互動率            1.2%     3.8%     +217% 🟢
客戶轉換率           0.8%     1.5%     +88% 🟢
品牌搜尋量           320/月   580/月   +81% 🟢
員工滿意度           3.2/5    3.6/5    +13% 🟡

── 歸因分析（定性） ──
營收提升主因：品牌定位釐清後 → 價格策略調整 + 目標客群聚焦
互動率提升主因：內容策略從「商品展示」轉為「故事敘事」

── 限制與聲明 ──
⚠️ 本報告為前後測對比，非嚴格因果推論。
外部因素（季節性、市場趨勢、競品變化）可能影響結果。
```

### 6.5 成效追蹤規則

- **誠實**：前後測是相關不是因果——報告必須聲明限制
- **客戶自報**：KPI 數據來自客戶，MUSEON 不做資料驗證
- **最小 KPI 集**：至少 3 個，不超過 8 個（太多追蹤不到）
- **基準線不可改**：建立後鎖定，避免回溯性偏差
- **歸因為定性**：用敘事說明可能原因，不做統計歸因

## 七、子系統六：Skill 使用率熱力圖（Phase 2 新增）

### 7.1 問題：26 個 Skill，哪些真的被用？

morphenix 要迭代，需要知道：哪些 Skill 高頻使用（值得深度優化）、哪些閒置（可能設計有問題或觸發條件太窄）、哪些被觸發但被跳過（可能誤觸率高）。

### 7.2 追蹤維度

| 維度 | 說明 |
|------|------|
| 觸發次數 | Skill 被 DNA27 路由觸發的次數 |
| 採納次數 | 觸發後實際被使用的次數 |
| 採納率 | 採納 / 觸發（低於 50% 可能有問題） |
| 深度使用 | 使用者在該 Skill 停留的輪次 |
| 共現頻率 | 與其他 Skill 一起出現的頻率（為 orchestrator 提供搭配依據） |
| 產業分布 | 不同產業客戶使用該 Skill 的比例 |

## 護欄

### 硬閘

**HG-EVAL-HONEST**：數據誠實。不美化、不過度歸因（相關≠因果）、退化必報告、樣本<10 標注「僅供參考」。

**HG-EVAL-CLIENT-DISCLAIMER**：客戶成效報告必須附限制聲明。前後測是相關不是因果，外部因素不可控。

**HG-EVAL-BASELINE-LOCK**：客戶基準線一旦建立不可回溯修改。

### 軟閘

**SG-EVAL-PRIVACY**：品質數據不含對話內容（只有分數和元資料）。

**SG-EVAL-SAMPLE**：Q-Score 依賴 AI 自評（有盲點）、滿意度是推估需偶爾校準、A/B 建議 ≥7 天觀察。

**SG-EVAL-USAGE-CONTEXT**：使用率低不等於 Skill 無用——需結合產業和場景判斷。

## 邊界宣告

**什麼情況下本 Skill 會失效**：
- 樣本太少（< 10 次互動）——統計無意義
- 客戶 KPI 造假或記憶偏差——垃圾進垃圾出
- 長期無 morphenix 迭代——追蹤數據沒有消費者就是浪費

**什麼問題不該用本 Skill 處理**：
- 即時品質控制（用 deep-think）
- 系統迭代決策（用 morphenix）
- 客戶的財務記帳和報稅（用外部工具）

**使用者最容易犯的錯**：
- 把 Q-Score 當絕對真理——它是 AI 自評，有系統性偏差
- 把客戶前後測當因果——只是相關性，必須附限制聲明
- 看到低分就恐慌——要看趨勢，不是看單次
