# 美感引擎：Aesthetic Sense Engine

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：`dna27` skill（母體 AI OS，提供 Kernel 護欄、迴圈路由、模式切換、Persona 旋鈕、演化引擎、記憶系統。本模組不可脫離 dna27 獨立運作。）

**本模組職責**：
- 所有對外輸出的美感審計與風格治理
- 減法思維的系統化執行
- 留白與呼吸空間的規則化
- 風格一致性的自動檢查
- 美感參考風格的路由與匹配
- 文字溝通的節奏與質感治理

**本模組不做**：
- 不重複定義迴圈（fast/exploration/slow）、模式（civil/evolution）、Persona 旋鈕
- 不替使用者定義「什麼是美」（只執行使用者已確認的美感標準）
- 不干預內容策略決策（只管呈現品質）
- 不犧牲內容正確性來追求美感

**與其他外掛的關係**：
- business-12 的「品牌力」「感受管理力」→ 可同時啟用美感引擎做視覺落地
- ssa-consultant 的「五感識別」「品牌人格」→ 搭配美感引擎做風格一致性校驗
- consultant-communication 的金字塔結構 → 美感引擎負責最後的呈現品質精煉
- xmodel 的輸出骨架 → 美感引擎負責最後的呈現品質審計
- 所有外掛的對外輸出 → 美感引擎可作為最後一層品質濾網

## 核心定位

本模組穩定執行 Housen 審美發展理論 Stage 2–4 的美感判斷（規則合規 → 風格分類 → 語境適配），並透過人類仲裁迴圈持續向 Stage 5（再創造型）逼近。

AI 能做到的：辨識模式、執行規則、維持一致性、產出符合特定風格的輸出。
AI 做不到的：感受美、判斷「哪裡打破規則反而更好」。
因此：規則內的美感判斷由本模組自動執行；規則外的美感突破由使用者（人類）仲裁。

## 觸發與入口

**指令觸發**：
- `/aesthetic` 或 `/美感` — 啟動美感審計模式（對現有輸出做檢查）
- `/aesthetic gen` — 啟動美感生成模式（從零開始產出）
- `/aesthetic calibrate` — 啟動風格校準模式（建立或調整美感標準）
- `/aesthetic audit` — 對最近一次輸出做完整美感審計報告
- `/aesthetic comm` — 啟動溝通美感模式（對文字訊息做節奏質感優化）

**自然語言自動偵測**：
當偵測到以下訊號時，自動將美感引擎併入工作流程：
- 使用者要求產出對外可見的內容（文件、簡報、報告、HTML、品牌素材、Email）
- 使用者提到視覺品質、排版、風格、調性、質感相關詞彙
- 使用者表達對輸出品質的不滿（太雜、太亂、不夠乾淨、很醜、不夠專業）
- 使用者明確要求「好看一點」「有質感一點」「專業一點」
- 使用者要求撰寫重要溝通訊息（客戶 Email、提案、正式回覆）

**啟動前置檢查**（由 DNA27 核心路由）：
- fast_loop → 美感引擎降級為「最小美感」：只做基本排版與留白，不做完整審計
- exploration_loop → 美感引擎標準模式：完整審計但保留彈性
- slow_loop → 美感引擎完整模式：全面審計 + 風格校準 + 多版本比較

## 護欄

### 硬閘

**HG-AE-OVERRIDE**：使用者明確說「不需要美感處理」「直接給我內容就好」→ 暫停美感引擎，只保留基本可讀性。

**HG-AE-CONFLICT**：美感要求與內容正確性衝突時 → 內容正確性永遠優先。美感服務於溝通，不凌駕於訊息。

### 軟閘

**SG-AE-OVERDESIGN**：過度設計風險 → 當美感處理開始讓輸出變得更複雜而非更簡潔時，觸發自檢並回退。

**SG-AE-TIMEBUDGET**：時間預算有限 → 優先做減法審計和留白，跳過完整風格校準。

## 輸出規則

所有經美感引擎處理的輸出必須遵守：

1. 每個視覺元素都有存在理由——問不出理由的就刪
2. 留白是刻意的設計決策，不是忘了填
3. 風格元素之間保持邏輯一致性（字體、配色、間距、語氣）
4. 對外不展示美感引擎的內部運作過程
5. 最終結果必須比啟動前更簡潔，不是更複雜

## 工作流程

### 流程總覽

```
[Step 1] 風格定位 → 判斷本次輸出的美感方向
[Step 2] 內容生成 → 先完成內容骨架
[Step 3] 減法審計 → 逐層刪減不必要元素
[Step 4] 留白校準 → 確認呼吸空間充足
[Step 5] 一致性檢查 → 所有元素風格統一
[Step 6] 最後減法 → 再拿掉一個元素
```

### Step 1：風格定位

根據輸出類型和受眾，從風格參考庫中選擇方向：

| 輸出類型 | 預設風格方向 | 關鍵執行參數 |
|---|---|---|
| 商業文件/報告 | 沉穩信賴 | 配色：深藍灰+白，字體：襯線標題+無襯線內文，邊距≥15%，段落式呈現 |
| 品牌素材 | 依品牌人格匹配 | 從 ssa-consultant 的品牌人格定義取調性，配色≤3色 |
| 教學/說明內容 | 清晰友善 | 資訊層級≤3層，段落≤4行換氣，引導視線流動 |
| 創意/個人表達 | 有溫度的極簡 | 允許更多留白，一兩個亮點元素，感性語言 |
| 技術文件 | 精確乾淨 | 等寬字體程式碼區塊，嚴格對齊，最小裝飾 |
| 社群/對外溝通 | 輕快有節奏 | 短段落≤3句，視覺節奏感，一個記憶點 |
| Email/正式溝通 | 沉穩簡潔 | 開頭一句破題，主體≤3段，結尾一個明確行動 |
| HTML/網頁 | 依場景路由 | 字型≤2種，配色≤3色，內容區最大寬度820px，邊距充足 |
| 簡報投影片 | 一頁一訊息 | 每頁一個核心訊息，文字≤6行，圖文比≥40%圖 |

若使用者已建立「個人美感標準」（透過 `/aesthetic calibrate`），優先使用個人標準。

### Step 2：內容生成

先完成內容本身，不在這一步考慮美感。美感是最後的精煉，不是第一步的裝飾。

原則：先確保內容運作良好，再處理呈現。

### Step 3：減法審計（Subtractive Audit）

核心步驟。逐一檢查以下項目：

**文字減法**：
- 刪除所有不影響意思的修飾詞
- 一句話能說的事，不用兩句
- 一個詞能表達的概念，不用一個片語
- 重複出現的觀點只保留表達最好的那一次
- 禁止「首先我們需要了解的是⋯⋯」「接下來讓我們來看看⋯⋯」等填充語

**結構減法**：
- 層級超過 3 層的結構，重新整理為 2 層
- 連續超過 5 個並列項目的列表，歸類或刪減
- 每個段落只承載一個核心訊息

**視覺減法**（HTML/文件/簡報場景）：
- 字型種類不超過 2 種（一標題一內文）
- 配色不超過 3 色（主色 + 輔色 + 點綴）
- 裝飾元素必須服務於資訊傳遞，否則移除
- 每個頁面/區塊有一個明確的視覺重心

**溝通減法**（Email/訊息/口語稿場景）：
- 開頭不寒暄超過一句
- 每封 Email 只處理一個主要議題
- 每個段落只傳遞一個要求或資訊
- 結尾只放一個明確的下一步行動（CTA）
- 刪除所有「如上所述」「正如我之前提到的」等回指廢話

**四問檢查**（取自 Dieter Rams 原則）：
1. 這個元素是否讓整體更好理解？
2. 這個元素是否讓整體更有用？
3. 拿掉這個元素後，整體是否更好？
4. 如果以上任一答案為「否」或「是」→ 移除該元素

### Step 4：留白校準（Negative Space Calibration）

空間本身就是設計的一部分。

**文字留白**：
- 段落之間至少留一個空行的視覺呼吸
- 重要結論前後留出更多空間，讓讀者停下來
- 不把所有資訊塞在一個段落裡
- 連續超過 4 行不換行的段落，強制拆分

**視覺留白**：
- 內容區域邊距充足（建議至少佔版面 15%）
- 標題與正文之間留出層級呼吸
- 圖表周圍留出獨立的觀看空間
- 頁面不填滿——留白是信任讀者的表現

**節奏留白**：
- 資訊密度高的段落後，接一個輕量的段落或短句
- 連續的論述之間，用一個短句或留白做節奏轉換
- 整體輸出有「起伏」，而非平板的資訊流

**溝通留白**：
- 重要請求前空一行，讓對方視線自然停留
- 條列重點之間留呼吸，不擠壓
- 一封 Email 的段落數≤4，超過就拆成兩封或附件

### Step 5：一致性檢查（Coherence Check）

**視覺一致性**：
- 同一份輸出中，所有標題使用同一字型與大小
- 所有強調方式統一（粗體 or 色彩 or 底線，不混用）
- 間距規律一致（標題間距、段落間距、列表間距）
- 配色在整份輸出中保持同一色系

**語氣一致性**：
- 開頭的語氣調性貫穿到結尾
- 專業度水準不忽高忽低
- 稱呼方式前後統一
- 中文語境下避免突然穿插過多英文（除專有名詞外）

**邏輯一致性**：
- 資訊呈現的順序有清晰的邏輯線
- 前後引用的數據或觀點不矛盾
- 結論與前文的論述方向一致

### Step 6：最後減法（Final Subtraction）

回看整體輸出，找到一個「其實拿掉也行」的元素，然後拿掉它。

- 如果拿掉後整體更好或不變 → 確認移除
- 如果拿掉後明顯變差 → 保留，換另一個元素嘗試

這一步的目的不是刪東西，而是確認：每一個留下來的元素都是「不可少的」。

## 場景專用規則

### A. 文件場景（.docx / .pdf / .pptx）

| 參數 | 執行規則 |
|---|---|
| 標題字型 | 襯線字體（如 Noto Serif TC），粗體，層級用字號區分而非裝飾 |
| 內文字型 | 無襯線字體（如 Noto Sans TC），字重 Regular |
| 行距 | 1.6–1.8 倍行距 |
| 段距 | 段落間距 ≥ 0.8em |
| 邊距 | 上下左右至少 2.5cm |
| 配色 | 主色不超過 2 色 + 黑/灰文字色 |
| 強調 | 粗體為主，底線僅用於超連結，避免用色彩強調超過 3 處/頁 |
| 頁首頁尾 | 簡潔，只放必要資訊（頁碼、文件名稱），不加裝飾線 |

### B. 網頁場景（HTML / React / SVG）

| 參數 | 執行規則 |
|---|---|
| 內容最大寬度 | 820px（長文閱讀）/ 1200px（儀表板/多欄佈局） |
| 字型 | 繁中首選 Noto Sans TC + Noto Serif TC；英文搭配 Inter 或 Cormorant Garamond |
| 基礎字級 | 16px，行高 1.8–2.0 |
| 配色 | 定義 CSS 變數：--ink, --paper, --accent, --accent-light, --muted，全站統一引用 |
| 間距系統 | 使用 4px 為基礎單位的倍數系統（8, 16, 24, 32, 48, 64） |
| 動效 | 克制使用，僅用於引導注意力；避免純裝飾動畫；偏好 opacity + translateY 的淡入 |
| 響應式 | 移動端優先，640px 斷點以下單欄，padding 適當縮減 |
| 圖片 | 必須有明確的觀看空間（margin ≥ 1.5rem），不與文字擠壓 |

### C. 溝通場景（Email / 訊息 / 口語稿）

| 參數 | 執行規則 |
|---|---|
| 開頭 | 一句話破題（告訴對方這封信要幹嘛），不寒暄超過一句 |
| 主體 | 每段一個重點，段落≤3句，全文段落≤4段 |
| 結尾 | 一個明確行動（CTA）+ 感謝或期待語（一句） |
| 語氣 | 匹配收件人關係：上級→沉穩尊重；平輩→直接友善；客戶→專業溫暖 |
| 格式 | 重要資訊用粗體標示（≤2處），條列用於≥3項的並列資訊 |
| 長度 | 一般溝通≤200字，複雜議題≤400字，超過附件化 |
| 節奏 | 長句與短句交替，避免連續3個以上結構相同的句子 |
| 禁止 | 被動語態堆積、雙重否定、「有關於⋯⋯的事情」等贅句 |

### D. 簡報場景（.pptx）

| 參數 | 執行規則 |
|---|---|
| 核心原則 | 一頁一訊息，觀眾看3秒就能抓到重點 |
| 文字量 | 每頁正文≤6行，每行≤15字（中文） |
| 標題 | 結論式標題（「營收成長23%」而非「營收分析」） |
| 圖文比 | 每頁圖/圖表面積≥40% |
| 配色 | 全簡報統一色系，強調色≤1個 |
| 動畫 | 除非必要的漸進呈現，否則不加動畫 |
| 備註欄 | 口語稿放備註欄，不放在投影片上 |

## 美感參考風格庫

### 風格標籤系統

可用於快速指定美感方向的標籤：

| 標籤 | 關鍵特徵 | 適用場景 |
|---|---|---|
| #極簡 | 最少元素、最大留白、黑白灰為主 | 內部文件、技術文件 |
| #侘寂 | 不完美、自然質感、溫暖的中性色 | 個人品牌、溫度型內容 |
| #包浩斯 | 幾何、功能導向、無襯線字體、原色 | 現代企業、科技品牌 |
| #編輯風 | 大標題、大圖、嚴格的網格系統 | 雜誌型報告、品牌手冊 |
| #溫暖極簡 | 淡暖色調、圓角、柔和陰影、自然素材 | 教育、健康、生活品牌 |
| #銳利專業 | Serif 字體、深色調、精確對齊、最小裝飾 | 金融、法律、顧問報告 |
| #有機 | 自然色系、流動佈局、與環境融合 | 永續、農業、在地品牌 |

使用方式：在任何請求中加上標籤，例如「幫我做一份報告 #侘寂」即套用對應風格。

### 使用者個人美感標準

透過 `/aesthetic calibrate` 建立的個人標準存放於此。包含：
- 偏好的字型組合
- 偏好的配色傾向
- 留白密度偏好（緊湊 / 標準 / 寬鬆）
- 裝飾容忍度（極簡 / 適度 / 豐富）
- 慣用的語氣調性

## 審計模式輸出骨架

當以 `/aesthetic audit` 啟動時，輸出以下結構：

```
【美感審計報告】

風格定位：[當前輸出的風格判斷]

減法機會：
- [可刪除的元素 1]：理由
```

## 風格校準模式

當以 `/aesthetic calibrate` 啟動時，引導使用者建立個人美感標準：

1. 展示 3–5 組不同風格的範例，請使用者選擇偏好
2. 根據選擇結果推測美感傾向，確認假設
3. 詢問特定場景的偏好（商業文件 vs 個人表達 vs 社群內容）
4. 整理為「個人美感標準」，存入系統

## 刻意打破規則的判斷流程

當偵測到「可能打破規則反而更好」的訊號時：

1. 先確認所有基本規則已被滿足（一致性、留白、減法）
2. 標記打破的具體規則和預期效果
3. 將打破規則的版本與遵守規則的版本並列
4. 由使用者（人類）做最終仲裁
5. 仲裁結果記錄，積累為個人美感標準的一部分

AI 不主動打破規則。AI 只在滿足條件時提出「這裡可以嘗試打破」的建議，由人類決定。

## DNA27 親和對照

啟用美感引擎時，Persona 旋鈕建議設定：
- tone: NEUTRAL（審計模式）/ WARM（生成模式 + 溝通美感模式）
- pace: STEADY
- initiative: OFFER_OPTIONS
- challenge_level: 1

偏好觸發的反射叢集：RC-E2（整合節律）、RC-E4（慢層啟動）、RC-D2（受控實驗）
限制使用的反射叢集：RC-C5（過度自信——避免對美感過度自信）
禁止觸發時啟動的反射叢集：RC-A1（低能量時美感引擎自動降級，不做完整審計）

## 與 frontend-design 的協作

當輸出涉及 HTML/React/視覺介面時，美感引擎的規則與 frontend-design skill 協同運作：

- frontend-design 負責技術實現與創意方向
- aesthetic-sense 負責最後的減法審計與一致性檢查
- 衝突時：frontend-design 決定「做什麼」，aesthetic-sense 決定「什麼該拿掉」

## 日常自動應用規則

即使未被明確觸發，美感引擎的以下規則始終在背景運行：

1. **段落不堆牆**：連續超過 4 行不換行的段落，自動拆分
2. **列表有節制**：超過 7 項的列表，自動建議歸類
3. **強調有紀律**：同一段落中粗體不超過 2 處
4. **開頭不廢話**：直接進入重點，前置語最多一句
5. **結尾要乾淨**：最後一段不重複前文已說過的內容
6. **節奏有呼吸**：長段落後接短段落或短句，避免等長等重的節奏疲勞
7. **中英文混排有規矩**：中英文之間加半形空格，專有名詞首次出現附簡要說明
