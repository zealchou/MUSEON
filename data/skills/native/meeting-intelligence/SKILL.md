# Meeting Intelligence：會議情報分析引擎

> **會議記錄是秘書的工作。會議情報是軍師的工作。**

---

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：`dna27` skill（母體 AI OS，提供 Kernel 護欄、迴圈路由、模式切換、Persona 旋鈕、演化引擎、記憶系統。本模組不可脫離 dna27 獨立運作。）

**本模組職責**：
- 接收會議逐字稿（文字或音檔），執行清理→結構化→多層分析→HTML 報告產出
- 第一層：結構化摘要（決議/行動項目/停車場）
- 第二層：圓桌多視角會診（含九策軍師動態召集）
- 第三層：博弈分析（陽謀佈局 + 陰謀人心）
- 第四層：人格動態分析（ANIMA 八原語推斷）
- 第五層：跨會議記憶整合（承諾漂移偵測 + 決策軌跡）
- 產出品牌級 HTML 戰情報告

**本模組不做**：
- 不做即時語音轉錄——那是 ASR 引擎（Whisper/Deepgram）的工作，本 Skill 接收轉錄後的文字
- 不做會議排程或行事曆管理
- 不替使用者做決策——只照亮會議中的動態，判斷權歸使用者
- 不做通用文字摘要（那是 text-alchemy 的工作）

**與其他外掛的關係**：

| 外掛 | 協作方式 |
|-----|---------|
| consultant-communication | 第一層結構化摘要用 SCQA + 金字塔原則 |
| roundtable | 第二層多視角會診，依會議類型動態召集角色 |
| master-strategy（九策軍師） | 第二層圓桌 + 第三層陽謀分析，軍師依會議類型動態召集 |
| shadow | 第三層陰謀分析，辨識防衛型發言、資訊不對稱、情緒勒索 |
| resonance | 第三層情緒轉折捕捉，標記共識/分歧/緊張時刻 |
| aesthetic-sense + design_spec.md | 第五層 HTML 報告品牌級排版 |
| knowledge-lattice | 第五層跨會議記憶，承諾漂移偵測 |
| eval-engine | 追蹤會議情報品質和使用者滿意度 |
| business-12 | 商業會議時辨識 12 力相關議題 |
| ssa-consultant | 銷售會議時標記顧問式銷售機會點 |

## 觸發與入口

**指令觸發**：
- `/meeting` — 啟動完整五層分析（預設根據會議類型動態調整深度）
- `/intel` — `/meeting` 的別名
- `/meeting quick` — 快速模式（只跑第一層結構化摘要，不含博弈分析）

- `/meeting status` — 查看目前在哪個階段
- `/meeting history` — 查看跨會議主題追蹤

**自然語言自動偵測**：
- 使用者貼上大段逐字稿或會議記錄
- 使用者說「幫我整理這段會議」「這場會議的重點是什麼」
- 使用者上傳音檔或語音訊息
- 使用者問「上次開會說的那件事後來怎樣了」（觸發跨會議記憶）

**啟動前置檢查**（由 DNA27 核心路由）：
- fast_loop → 快速模式（第一層結構化摘要 only）
- exploration_loop → 標準模式（第一～三層）
- slow_loop → 完整模式（五層全開，含 HTML 報告）

## 核心工作流程：七階段管線

```
[Stage 0] 收錄 — 接收逐字稿/音檔，判斷會議類型
    ↓
[Stage 1] 清理 — 去贅字、加標點、修正不完整句
    ↓
[Stage 2] 結構化 — 說話者辨識、主題分段、時間戳對齊
    ↓
[Stage 3] 結構化摘要 — SCQA 框架、決議/行動項目/停車場
    ↓
[Stage 4] 深度分析 — 圓桌會診 + 博弈分析 + 人格動態
    ↓
[Stage 5] 記憶整合 — 跨會議追蹤、承諾漂移偵測
    ↓
[Stage 6] 報告產出 — 品牌級 HTML 戰情報告
```

### Stage 0：收錄（Ingest）

**目標**：接收輸入，判斷會議類型，決定分析深度和軍師召集名單。

**支援的輸入格式**：
- 純文字逐字稿（直接貼入或上傳 .txt/.md）
- 其他會議工具的匯出（Plaud/Otter/Fireflies/雅婷逐字稿 格式自動辨識）
- 使用者自己的會議筆記（Granola 式：粗略筆記 + AI 增強）

**會議類型自動辨識**：
| 會議類型 | 辨識關鍵詞 | 分析深度 | 軍師召集 |
|---------|-----------|---------|---------|
| 商業談判 / 合作洽談 | 合約、條件、報價、合作、MOU | 五層全開 | 鬼谷子（捭闔）+ 孫子（虛實）+ 司馬法（治理） |
| 內部策略會議 | 策略、方向、目標、OKR、roadmap | 五層全開 | 孫子（全勝）+ 六韜（組織）+ 將苑（領導） |
| 銷售會議 / 客戶拜訪 | 客戶、需求、報價、成交、異議 | 五層全開 | 鬼谷子（揣摩）+ 三略（上中下策）+ 尉繚子（賞罰） |
| 危機處理 / 問題解決 | 問題、緊急、風險、損失、修復 | 五層全開 | 孫子（速戰）+ 吳子（應變）+ 唐李問對（實戰） |
| 投資 / 財務會議 | 營收、成本、ROI、融資、估值 | 五層全開 | 三略（資源）+ 司馬法（紀律）+ A3（現實顯化） |
| 日常站會 / 進度同步 | 進度、卡點、下一步、sprint | 三層（跳過博弈） | 不召集軍師 |
| 腦力激盪 / 創意發想 | 點子、方案、想法、如果 | 三層 + xmodel | 不召集軍師，改用 xmodel 破框 |

**輸出物**：會議元資料卡
```
會議類型：[自動辨識 / 使用者指定]
參與者：[從逐字稿辨識]
時長：[估算]
分析深度：[快速 / 標準 / 完整]
軍師召集：[依類型動態決定]
```

**驗證**：會議類型由使用者確認後才進入 Stage 1。使用者可覆寫自動判斷。

### Stage 1：清理（Clean）

**目標**：將口語逐字稿轉化為可讀的書面文字，保留語意完整性。

**方法**（LLM Prompting，不需額外模型）：

**1.1 去贅字**（台灣中文特化三層分類）：
| 分類 | 處理方式 | 範例 |
|------|---------|------|
| **純贅字**（移除） | 刪除，不影響語意 | 「嗯...那個...就是說...」→ 刪除 |
| **語氣標記**（轉換） | 轉為書面語氣標注 | 「齁，這樣不太對吧」→「（質疑語氣）這樣不太對吧」 |
| **確認語**（保留） | 保留，表示共識信號 | 「對，我同意」→ 保留（共識標記） |

**1.2 重複合併**：
- 「我我我覺得」→「我覺得」
- 「就是就是」→「就是」

**1.3 不完整句修正**：
- 「往左邊...不對往右邊走」→「往右邊走」
- 保留修正痕跡作為博弈分析素材（猶豫 = 可能的內在衝突信號）

**1.4 中英夾雜處理**：
- 專有名詞保留英文原形：API、AWS、Sprint、KPI
- 非必要英文轉中文：「concern 的 point」→「擔心的重點」
- 晶晶體保留但標記：「這個 case 很 tricky」→「這個案例很棘手（原：case/tricky）」

**1.5 標點與斷句**：
- 加入逗號「，」句號「。」問號「？」
- 連續超過 50 字無標點 → 強制斷句
- 使用全形標點

**輸出物**：清理後逐字稿（保留說話者標記 [Speaker X]）

**驗證**：抽查 3 段，確認語意未被扭曲。

### Stage 2：結構化（Structure）

**目標**：為清理後的逐字稿加上結構——說話者、主題、時間。

**方法**：

**2.1 說話者辨識**：
- 若逐字稿已有說話者標記 → 直接使用
- 若無 → 依發言風格、稱呼線索、上下文推斷（LLM 能做到 85%+ 準確度）
- 標記格式：`【人物A】`、`【人物B】` 或使用者指定的真名

**2.2 主題分段**（LLM 語意分段）：
- 依語意轉折切分為 topic blocks
- 每個 block 一個主題標題（3-8 字）
- 標記 block 之間的轉場方式（自然過渡 / 被打斷 / 主持人引導）

**2.3 時間戳對齊**（如有）：
- 音檔轉錄的時間戳保留
- 純文字逐字稿估算「第 N 段 / 全文 M%」

**輸出物**：結構化逐字稿（topic blocks × speaker × timeline）

### Stage 3：結構化摘要（Summarize）— 第一層

**目標**：產出專業級結構化會議摘要。調用 **consultant-communication** 的 SCQA + 金字塔原則。

**摘要結構**：

```markdown
## 執行摘要（2-3 句）
[核心結論 + 最重要的決議 + 最急迫的行動]

## 討論主題
### 主題一：[標題]
- 討論要點（bullet points，保留關鍵細節）
- 關鍵引述：「...」—— 【人物A】
- 分歧意見：【人物B】認為...，【人物C】認為...

### 主題二：[標題]
...

## 決議事項
| # | 決議內容 | 提出者 | 共識程度 |
|---|---------|--------|---------|
| 1 | ... | 【人物A】 | 全體同意 / 多數同意 / 有保留 |

## 行動項目
| # | 任務 | 負責人 | 期限 | 來源主題 |
|---|------|--------|------|---------|
| 1 | ... | 【人物A】 | YYYY-MM-DD | 主題一 |

## 停車場（離題但值得後續討論）
- [議題] — 建議在 [場合] 討論

## 下次會議建議
- 日期：[建議]
- 優先議程：[從本次未完成事項推導]
```

**輸出物**：結構化摘要（Markdown）

**驗證**：每個決議都能追溯到逐字稿中的具體發言。

### Stage 4：深度分析（Deep Analysis）— 第二～四層

**目標**：超越「發生了什麼」，分析「背後發生了什麼」。此階段是 MUSEON 獨有的價值。

#### 第二層：圓桌多視角會診

調用 **roundtable** + **master-strategy 九策軍師**，依 Stage 0 的軍師召集名單動態組成。

**圓桌結構**：
```markdown
## 🏛️ 圓桌會診

### 軍師視角

**[軍師名 1]（[流派]）的觀點**：
「從 [角度] 來看，這場會議的關鍵轉折在 [事件]。
 [人物A] 的做法相當於 [兵法類比]，但忽略了 [風險]。」
→ 建議：[一句話]

**[軍師名 2]（[流派]）的觀點**：
「[軍師 1] 說得對，但 [反面觀點]...」
→ 建議：[一句話]

**[軍師名 3]（[流派]）的觀點**：
「你們都忽略了 [盲點]...」
→ 建議：[一句話]

### 觀點衝突
- 軍師 1 vs 軍師 2：[分歧點] — 你需要決定 [選擇]
- 共識點：[所有軍師都同意的]
```

#### 第三層：博弈分析

**陽謀分析**（master-strategy）：
```markdown
## ⚔️ 陽謀佈局分析

### 權力動態
- 誰主導了議程？[人物 + 證據]
- 誰在試探？[發言模式分析]
- 誰在讓步？是真讓步還是以退為進？

### 戰略意圖
- [人物A] 的佈局方向：[推斷 + 證據]
- [人物B] 的佈局方向：[推斷 + 證據]
- 時機判斷：[現在是攻還是守的時刻？]
```

**陰謀分析**（shadow）：
```markdown
## 🌑 陰謀辨識

### 防衛型發言偵測
- [人物C] 在 [主題X] 上多次迴避直接回答 → 可能的隱性抗拒
- [人物D] 的措辭從「我們應該」轉為「你們可以」→ 責任切割信號

### 資訊不對稱
- [人物A] 掌握了 [某資訊] 但選擇不在會議中公開 → 推斷依據：[...]

### 情緒勒索 / 操控模式（如偵測到）
- 類型：[shadow 27 類型中的哪一種]
- 證據：[具體發言]
- 防禦建議：[一句話]
```

**情緒轉折**（resonance）：
```markdown
## 🌊 情緒動態

### 情緒地圖
| 時間點 | 事件 | 情緒轉折 | 觸發者 |
|--------|------|---------|--------|
| 主題二開始 | 預算討論 | 緊張感上升 | 【人物B】提出刪減 |
| 主題二中段 | 妥協方案 | 緩和 | 【人物A】讓步 |
| 主題三 | 人事異動 | 沈默（壓抑） | 全場 |

### 共識 vs 分歧時刻
- 🟢 真共識：[具體時刻 + 語言證據]
- 🔴 假共識（表面同意但...）：[具體時刻 + 微表情/措辭證據]
```

#### 第四層：人格動態分析（ANIMA 八原語）

```markdown
## 👤 參與者人格動態

### 【人物A】
- 推斷原型：[八原語中的哪一型]
- 發言模式：[主導型 / 分析型 / 調和型 / 觀察型]
- 本次會議角色：[議程推動者 / 問題提出者 / 沈默觀察者]
- 溝通建議：下次與此人互動時，[具體建議]

### 【人物B】
...

### 人際動態圖
- 【A】↔【B】：[同盟 / 對立 / 試探]
- 【C】→【D】：[單向影響 / 依賴]
```

**輸出物**：深度分析報告（第二～四層）

**驗證**：每個推斷都標注「證據強度」（強/中/弱）和具體發言引用。

### Stage 5：記憶整合（Memory Integration）— 第五層

**目標**：將本次會議與歷史記憶交叉比對，發現承諾漂移和主題演進。

**方法**：調用 **knowledge-lattice** 搜尋相關會議結晶。

```markdown
## 🧠 跨會議記憶

### 承諾漂移偵測
| 承諾內容 | 承諾時間 | 承諾人 | 本次狀態 |
|---------|---------|--------|---------|
| 「Q2 前完成 XX」 | 2026-02-15 | 【人物A】 | ⚠️ 未提及 |
| 「下週提供報價」 | 2026-03-20 | 【人物B】 | ✅ 已完成 |
| 「預算增加 20%」 | 2026-03-01 | 【人物C】 | 🔴 今天改口為 10% |

### 主題演進追蹤
- [主題X] 在最近 3 次會議中的演進：
  - 3/1：初步提出 → 3/15：方案討論 → 今天：決議執行
  - 趨勢：[收斂中 / 發散中 / 停滯]

### 決策軌跡
- 本次新增決策：[N] 項 → 已寫入 knowledge-lattice
- 本次變更決策：[M] 項 → 標記變更理由
```

**輸出物**：跨會議記憶報告

**驗證**：承諾漂移的偵測結果需與 knowledge-lattice 中的歷史結晶比對一致。

### Stage 6：報告產出（Deliver）

**目標**：將五層分析組裝成品牌級 HTML 戰情報告。

**方法**：
1. 讀取 `data/_system/brand/design_spec.md` 確保品牌一致性
2. 調用 aesthetic-sense 做最終排版審計
3. 生成多版本輸出

**輸出版本**：
| 版本 | 內容 | 適用場景 |
|------|------|---------|
| **完整戰情報告**（HTML） | 五層全含，含摺疊面板 | 存檔、深度回顧 |
| **執行摘要**（1 頁） | 第一層精華 + 行動項目 | 會後立即分享 |
| **行動項目清單**（純列表） | 只有待辦 + 負責人 + 期限 | 直接轉任務系統 |

**HTML 報告結構**：
```
📊 Meeting Intelligence Report
├── Header：會議元資料（類型/日期/參與者/時長）
├── Section 1：執行摘要（常開）
├── Section 2：討論主題 + 決議 + 行動項目（常開）
├── Section 3：圓桌會診 + 軍師觀點（摺疊）
├── Section 4：博弈分析——陽謀 + 陰謀 + 情緒（摺疊）
├── Section 5：人格動態（摺疊）
├── Section 6：跨會議記憶（摺疊）
└── Footer：生成時間 + MUSEON 品牌標記
```

**驗證**：HTML 報告必須通過 aesthetic-sense 的減法審計 + 一致性檢查。

| 4,000-50,000 tokens | Sliding Window：每 chunk 3,000 tokens + 500 overlap |

## 護欄

### 硬閘

**HG-INTEL-EVIDENCE**：所有推斷必須標注證據強度和具體發言引用。沒有證據的分析 = 不輸出。

**HG-INTEL-NO-FABRICATE**：不可捏造會議中未發生的事。推斷和事實必須明確區分，推斷用「可能」「推測」標記。

**HG-INTEL-PRIVACY**：博弈分析和人格分析僅供會議主持人/決策者參考，不應直接分享給被分析者。報告首頁附隱私聲明。

**HG-INTEL-BALANCE**：每個人的分析必須平衡——不可只分析一方的「陰謀」而忽略另一方。

### 軟閘

**SG-INTEL-DEPTH-MATCH**：日常站會不需要五層分析。會議類型與分析深度必須匹配，避免過度分析。

**SG-INTEL-SPECULATION-LIMIT**：博弈分析的推測部分不超過事實部分的 30%。推測比例過高時自動降級為「觀察」而非「分析」。

**SG-INTEL-PARTICIPANT-MINIMUM**：2 人以下的對話不觸發圓桌會診和人格動態分析（因為角度不夠多）。

### 輸出規則

1. 每個決議追溯到逐字稿中的具體發言
2. 每個推斷標注「證據強度」：強（多處發言佐證）/ 中（單處發言）/ 弱（語氣/沈默推斷）
3. 博弈分析用中性語言，不下道德判斷
4. 人格分析基於發言模式，不基於刻板印象
5. 跨會議比對只引用 knowledge-lattice 中有記錄的歷史

## 邊界宣告

**什麼情況下本 Skill 會失效**：
- 逐字稿品質太差（辨識率 < 70%）——垃圾進垃圾出
- 單人獨白（演講/podcast）——沒有人際動態可分析，改用 text-alchemy 做摘要
- 使用者要求「客觀記錄」但不要任何分析——改用 `/meeting quick`

**什麼問題不該用本 Skill 處理**：
- 通用文字摘要（用 text-alchemy）
- 演講/podcast 逐字稿整理（用 text-alchemy + c15）
- 法庭/仲裁等法律場景的逐字稿（需法律專業，超出本 Skill 範圍）

**使用者最容易犯的錯**：
- 把博弈分析當「真相」——那是基於語言模式的推斷，不是心理測量
- 把人格分析結果拿去「對質」被分析者——那是戰略參考，不是指控工具
- 日常站會也要求五層分析——過度分析比不分析更浪費時間
