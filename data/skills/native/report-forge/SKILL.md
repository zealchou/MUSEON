# Report-Forge：付費級產業診斷報告鍛造引擎

> **ChatGPT 給你選項，我們給你判斷。ChatGPT 給你通用，我們給你你的。**

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：dna27 skill（母體 AI OS）

**本模組職責**：
- 定義「付費級報告」與「免費 AI 回答」的方法論差距
- 執行七層報告結構（結論先行 → MECE 拆解 → 多視角 → 案例 → 情境模擬 → 行動清單 → 暗示擴展）
- 執行三角驗證方法論（資料 × 方法論 × 研究者 × 理論）
- 產出精美 HTML 報告（遵循 MUSEON 品牌設計規範）
- 管理報告的深度控制（展示版 vs 完整版）

**本模組不做**：
- 不做金融市場多空研判（那是 market-core 的工作）
- 不做商業模式框架診斷（那是 business-12 的工作）
- 不做溝通結構教學（那是 consultant-communication 的工作）
- 不捏造案例或數據——搜不到就標「數據待驗證」
- 不替客戶做最終決策——只輸出診斷與路徑

**協作矩陣**：

| 外掛 | 協作方式 |
|-----|---------|
| dse | 方法論來源——三角驗證、第一性原理拆解的工程方法論由 DSE 提供 |
| consultant-communication | 結構工具——金字塔原則、MECE、SCQA、HBR 8S 從此處調用 |
| business-12 | 框架工具——12 力商業診斷框架作為 Layer 2 的分析底板 |
| market-core | 市場數據——需要市場規模、趨勢數據時調用五層蒐集引擎 |
| master-strategy | 戰略推演——需要多方案博弈分析時調用 |
| aesthetic-sense | 美感審計——HTML 報告產出前調用 |
| storytelling-engine | 案例敘事——嵌入式案例的敘事結構調用 |
| deep-think | 全程常駐品質審計 |

## 護欄

### 硬閘（觸發時強制執行）

**HG-RF-TRIANGULATION**：每個關鍵論點必須經過至少兩個獨立來源的交叉驗證。單一來源的論點只能標記為「待驗證假設」，不得作為報告結論的支撐。

**HG-RF-NO-FABRICATION**：禁止捏造案例、數據、來源。搜不到真實案例時，標記為「案例待補充」或使用匿名化的「某品牌 A」並註明「基於產業通則推演」。

**HG-RF-SOURCE**：所有數據必須附來源。無來源的數據標記為「估算值」並說明估算邏輯。

**HG-RF-BRAND**：HTML 報告必須讀取 `~/MUSEON/data/_system/brand/design_spec.md` 後產出，遵循品牌規範。

### 軟閘（觸發時限制範圍）

**SG-RF-DEPTH**：根據報告模式調整深度——展示版（5-8 頁，刻意留缺口引發需求）vs 完整版（15-20 頁，可直接用於決策）。

**SG-RF-CASE-QUALITY**：案例品質分三級——Tier 1（同產業同規模真實案例）> Tier 2（同產業不同規模）> Tier 3（跨產業類比）。優先使用 Tier 1。

## 核心方法論：為什麼值得付費？

### 免費 AI 回答 vs 付費級報告的六大差距

| 維度 | ChatGPT 能做 | Report-Forge 做的 |
|------|-------------|-------------------|
| **資料源** | 訓練資料裡的通用知識 | 3+ 獨立來源交叉驗證同一個數據點 |
| **分析結構** | 「可以考慮 A 也可以考慮 B」 | MECE 拆解後告訴你「先做 A，因為⋯⋯」 |
| **案例** | 泛用案例（Apple、Tesla） | 嵌入式案例——同產業、同規模的真實故事 |
| **多視角** | 單一「顧問」口吻 | 老闆看什麼、財務看什麼、營運看什麼 |
| **預測** | 線性外推 | 三情境模擬（樂觀/基準/悲觀）+ 敏感度分析 |
| **行動性** | 「建議優化客戶留存率」 | 「你的產業回購週期 6 週，這週做這件事：⋯⋯」 |

### 三角驗證方法論（Triangulation）

> 來源：DSE 第一性原理拆解 → 國際付費報告（McKinsey / Forrester / CB Insights）的核心方法論萃取

四種三角驗證形式：

**1. 資料三角驗證（Data Triangulation）**
- 同一個數據點用至少兩種來源驗證
- Top-down（產業總產值推算）× Bottom-up（個別公司加總推算）
- 兩數必須收斂，差距 > 30% 時標記為「數據分歧」

**2. 方法論三角驗證（Methodological Triangulation）**
- 量化數據（市場規模、成長率、毛利率） × 質性分析（案例、訪談、觀察）
- 數據說「市場在成長」但案例說「大家都在虧」→ 標記矛盾，深挖原因

**3. 研究者三角驗證（Investigator Triangulation）**
- 同一份資料從三個角色視角解讀：老闆（戰略）、CFO（財務）、COO（營運）
- 三個角色的結論不一致 → 報告必須呈現分歧，不強行統一

**4. 理論三角驗證（Theory Triangulation）**
- 用至少兩種分析框架看同一產業
- 例：Porter's Five Forces + 價值鏈分析 + business-12 的商業模式框架
- 不同框架給出不同結論 → 報告呈現多元觀點

## 七層報告結構

### Layer 1：結論先行（McKinsey SCR 框架）

> 調用 consultant-communication 的金字塔原則 + SCQA

- **Situation**：你所在的產業現在怎麼了（1-2 段，建立共識）
- **Complication**：為什麼老方法行不通了（1-2 段，製造張力）
- **Resolution**：你該怎麼辦（1-3 句核心建議，金字塔頂層）

**執行規則**：
- 結論必須有立場（「你應該先做 A」），不是列選項
- 結論背後的邏輯在後續層展開
- 讀者讀完 Layer 1 就知道報告的核心訊息

### Layer 2：MECE 深度拆解

> 調用 consultant-communication 的 MECE 原則 + business-12 的診斷框架

把核心問題切成 4-6 個互不重疊、合起來完整的子問題。

每個子問題包含：
1. **數據事實**（附來源連結）
2. **交叉驗證**（至少兩個獨立來源，標記一致/矛盾）
3. **第一性原理拆解**（為什麼是這樣？限制條件是什麼？）
4. **白話翻譯**（專有名詞解釋，「你媽能聽懂」標準）

**MECE 切法選擇**：
- 財務型問題 → 公式法（利潤 = 收入 - 成本）
- 流程型問題 → 流程法（獲客 → 轉換 → 留存 → 推薦）
- 能力型問題 → 要素法（人/財/物/資訊）或 business-12 框架

### Layer 3：多視角切入

同一份數據，從三個角色視角分別解讀：

| 視角 | 關注焦點 | 語言風格 |
|------|---------|---------|
| **老闆/創辦人** | 市場方向、品牌定位、競爭策略 | 戰略語言、願景導向 |
| **財務長/會計** | 毛利結構、現金流、ROI、損益平衡 | 數字語言、風險導向 |
| **營運長/店長** | 人力配置、流程效率、客戶體驗、可擴展性 | 執行語言、效率導向 |

**執行規則**：
- 三個視角可能給出矛盾的建議（例：老闆要擴張，CFO 說現金流不夠）
- 矛盾必須呈現，不強行統一——這正是報告的價值
- 每個視角的結論用 1-2 句摘要

### Layer 4：嵌入式案例

> 調用 storytelling-engine 的敘事結構

每個關鍵論點後面接一個案例。案例是論證的一部分，不是附錄裡的故事。

**案例結構**（A-B-T 極簡敘事）：
```
[品牌名] 在 [時間] 面臨 [情境]，而且 [正面條件]。
但是 [轉折/挑戰]。
因此 [他們做了什麼/結果如何]。
```

**案例品質分級**：
- **Tier 1**：同產業、同規模（月營收 100-500 萬）的台灣真實案例 → 最有說服力
- **Tier 2**：同產業但不同規模（大企業或微型企業）→ 需說明規模差異
- **Tier 3**：跨產業類比 → 需說明為什麼可類比

**案例配對原則**：成功案例 + 失敗案例配對呈現。只有成功案例 = 倖存者偏差。

**案例搜尋策略**：
1. web_search：搜「[產業] [台灣/亞洲] 案例 成功/失敗 [年份]」
2. web_search：搜「[產業] case study SMB [region]」
3. 搜不到 Tier 1 → 降級到 Tier 2 → 降級到 Tier 3
4. 全都搜不到 → 標記「案例待補充」，用匿名化推演

### Layer 5：情境模擬

三個情境，每個都有具體數字：

| 情境 | 假設條件 | 預估影響 | 機率判斷 |
|------|---------|---------|---------|
| **樂觀** | 如果你做對了這些事⋯⋯ | 毛利提升 X%，營收成長 Y% | 條件滿足時 Z% |
| **基準** | 維持現狀⋯⋯ | 毛利持平/微降，市場份額被侵蝕 | 預設路徑 |
| **悲觀** | 如果忽略這個問題⋯⋯ | 毛利下滑 X%，可能在 Y 個月內⋯⋯ | 不行動時 Z% |

**執行規則**：
- 數字必須有來源或推算邏輯
- 基準情境 = 「什麼都不做」的自然演化
- 樂觀和悲觀情境必須附「敏感度分析」——哪個變數最能改變結果？

### Layer 6：行動清單

不是「建議做 X」，而是精確到可執行的步驟：

**這週做什麼（0-7 天）**：
- 具體步驟（誰做、做什麼、怎麼做）
- 預期產出物（可驗證的交付物）
- 所需資源（時間、工具、人力）

**這個月做什麼（7-30 天）**：
- 結構同上
- 與「這週」的因果連結

**三個月後檢查什麼指標**：
- 哪些 KPI 應該有變化
- 變化幅度的預期範圍
- 如果沒變化，代表什麼（診斷下一步）

### Layer 7：暗示擴展（展示版限定）

報告結尾暗示「還有哪些主題可以深入」：

> 「本報告聚焦 [主題 A]。延伸主題：[主題 B]、[主題 C]、[主題 D]⋯⋯」

**設計原則**：
- 展示版：Layer 7 是必要的（引發客戶需求）
- 完整版：Layer 7 可選（客戶已付費，不需再勾引）
- 暗示的主題必須是 MUSEON 真正能交付的（不畫餅）

## 報告生產工作流程

### Phase 0：任務定錨

開始前必須確認：
1. **目標產業**：什麼產業？
2. **目標客戶畫像**：月營收規模、經營痛點、發展階段
3. **報告模式**：展示版（5-8 頁）/ 完整版（15-20 頁）
4. **交付格式**：HTML（預設）/ Markdown / PDF

### Phase 1：資料蒐集（三角驗證）

**搜尋輪次**：6-10 輪 web_search

**搜尋維度**：
1. 產業規模與趨勢（top-down + bottom-up 交叉驗證）
2. 毛利結構與成本結構（同產業 3+ 來源比對）
3. 競爭格局與進入障礙
4. 客戶行為與消費趨勢
5. 技術變革與數位化程度
6. 同產業案例蒐集（成功 + 失敗各 2-3 個）

**資料品質檢查**：
- 來源日期 < 2 年 → 有效
- 來源日期 2-5 年 → 標記「數據可能已過時」
- 來源日期 > 5 年 → 不使用

### Phase 2：分析與結構化

1. 用 MECE 切割核心問題
2. 每個子問題執行三角驗證
3. 從三個視角分別解讀
4. 嵌入搜尋到的案例
5. 建構三情境模擬

### Phase 3：報告撰寫

按七層結構依序撰寫。每層寫完後檢查：
- [ ] 是否有立場（不是列選項）？
- [ ] 是否有數據支撐（附來源）？
- [ ] 是否有白話翻譯？
- [ ] 是否通過 HBR 8S 審計？

### Phase 4：HTML 產出

1. 讀取 `~/MUSEON/data/_system/brand/design_spec.md`
2. 調用 aesthetic-sense 美感審計
3. 產出 HTML 報告

**HTML 結構**：
1. Hero Section — 報告標題、產業、日期
2. Executive Summary — Layer 1 結論先行
3. Deep Dive — Layer 2-3（MECE 拆解 + 多視角）
4. Case Studies — Layer 4 嵌入式案例
5. Scenarios — Layer 5 情境模擬
6. Action Plan — Layer 6 行動清單
7. What's Next — Layer 7 暗示擴展（展示版限定）
8. Footer — MUSEON 品牌 + 免責聲明 + 資料截止日期

### Phase 5：品質審計

最終檢查清單：
- [ ] 三角驗證：關鍵論點是否有 2+ 獨立來源？
- [ ] 案例品質：是否有同產業案例？成功失敗配對？
- [ ] 多視角：三個角色是否都有獨立解讀？
- [ ] 行動性：行動清單是否具體到「這週做什麼」？
- [ ] 白話：所有專有名詞是否附解釋？
- [ ] 品牌：是否符合 MUSEON 設計規範？

## 產業報告主題設計模板

針對不同產業，報告主題的設計邏輯：

| 產業 | 報告主題公式 | 範例 |
|------|------------|------|
| 服務業（美業/餐飲） | 「[產業] [年份]：月營收 [X-Y] 萬的 [痛點數] 大 [痛點類型] 與突圍路徑」 | 「台灣美業 2026：月營收 100-300 萬門店的三大毛利殺手與突圍路徑」 |
