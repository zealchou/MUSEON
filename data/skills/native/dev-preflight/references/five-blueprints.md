# Dev-Preflight：開發前置飛行檢查

> **飛機起飛前，機長不管飛了多少次，每次都要跑一遍 preflight checklist。**
> **不是因為不信任自己，是因為系統性遺漏靠記憶力防不住。**

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：`dna27` skill（母體 AI OS）

**本模組職責**：
- 在開發指令下達前，用白話提問覆蓋五張架構藍圖
- 根據任務規模自動分級（快速版 / 標準版 / 完整版）
- 載入歷史教訓（來自 dev-retro 的結晶）做情境比對
- 產出「藍圖摘要卡」作為 Claude Code 的上下文輸入
- 把工程師「理所當然知道」但非工程師「根本不知道要問」的東西挖出來

**本模組不做**：
- 不寫程式碼（那是 Claude Code 的工作）
- 不做完整的架構設計文件（那是專業架構師的工作）
- 不做開發後的品質審計（那是 qa-auditor 的工作）
- 不做開發後的經驗回溯（那是 dev-retro 的工作）
- 不做技術可行性研究（那是 dse 的工作）

## 觸發與入口

**指令觸發**：
- `/preflight` — 啟動標準版（自動判斷任務規模）
- `/pf` — `/preflight` 的簡寫
- `/pf quick` — 強制快速版（2-3 題）
- `/pf full` — 強制完整版（五張圖全展開）
- `/pf history` — 顯示歷史教訓中與當前任務相關的紀錄

**自然語言自動偵測**：
- 使用者描述要建立新功能或新模組
- 使用者描述要把 A 跟 B「接起來」或「整合」
- 使用者描述要重構或大幅修改現有系統
- 使用者要對 Claude Code 下達涉及多檔案的開發指令

## 核心理念

### 問題本質

非工程師用 AI 寫程式時，最大的風險不是「程式寫錯」——那 AI 通常能修。
最大的風險是**認知盲區**：你不知道要問什麼，所以 AI 也不會主動告訴你。

工程師花四年學校加五年實戰才內化的東西，包括：
- 系統之間不會「自然」連在一起，需要明確定義介面
- 資料格式不約定好，兩邊講不同語言
- 一個壞了其他怎麼辦，不是「不會壞」而是「壞了要怎麼處理」
- 「能跑」跟「穩定跑」是兩回事
- 現在能跑但三個月後要改會痛死

這些不是你「笨」或「不努力」——是**領域知識的結構性缺口**。
Dev-Preflight 就是把這個缺口系統性地補上。

### 設計哲學

**不教你寫程式，教你問對問題。**

用建築工程的五張藍圖概念，把軟體架構的核心關注點翻譯成你能回答的白話提問。
你回答完，系統自動產出一張「藍圖摘要卡」，餵給 Claude Code 當上下文。

## 五張藍圖體系

### 理論基礎

融合兩個經典軟體架構模型，翻譯成非工程師友善的版本：

| 經典模型 | 我們的白話版 | 建築工程對應 |
|---------|------------|------------|
| 4+1 邏輯視圖 + C4 Context/Container | 📐 功能地圖 | 建築平面圖 |
| 4+1 流程視圖 + 拓撲圖 | 🕸️ 連線地圖 | 水電管線圖 |
| 4+1 開發視圖 + C4 Component | 📦 分工地圖 | 施工分工圖 |
| 4+1 部署視圖 | 🖥️ 部署地圖 | 結構配置圖 |
| 4+1 場景視圖 + C4 Code walkthrough | 🎬 走一遍 | 動線模擬 / 爆炸圖 |

### 📐 藍圖一：功能地圖（這個東西管什麼事？）

**對應工程概念**：邏輯視圖、系統邊界、職責劃分、依賴關係
**建築比喻**：這棟樓有幾個房間？各房間管什麼事？牆在哪？

**完整版提問（5 題）**：

1. **功能邊界**：用一句話說，這個功能做完之後，使用者能做到什麼以前做不到的事？
2. **職責劃分**：這個功能只管自己的事，還是它需要「知道」其他模組的存在？如果需要，列出來。
3. **輸入輸出**：它吃什麼進去（資料、指令、事件）？吐什麼出來？
4. **不做清單**：明確說出這個功能「不負責」什麼。（這題最容易漏，也最重要）
5. **依賴方向**：如果把這個功能拿掉，其他哪些東西會壞？反過來，它壞了的時候希望其他東西怎麼反應？

**快速版提問（2 題）**：
1. 這個功能做什麼、不做什麼？（邊界）
2. 它需要跟誰互動？（依賴）

### 🕸️ 藍圖二：連線地圖（誰跟誰要說話？）

**對應工程概念**：拓撲、介面契約、訊息格式、通訊協議、錯誤處理
**建築比喻**：水管從哪接進來、經過哪些管子、到哪裡出去？接頭規格是什麼？漏水怎麼辦？

> ⚠️ **這是非工程師最常踩坑的藍圖。**
> 「我以為子系統會自然運作在一起」——不會。每條連線都需要明確約定。

**完整版提問（7 題）**：

1. **連線清單**：列出所有「A 需要跟 B 說話」的組合。不確定的也列。
2. **訊息格式**：每條連線上傳的是什麼？文字？JSON？事件？有沒有約定格式？
3. **方向性**：每條連線是單向（A 發 B 收）還是雙向（A 問 B 答）？
4. **時序**：這些連線有先後順序嗎？A 必須先完成 B 才能開始？還是可以同時跑？
5. **斷線處理**：如果某條連線斷了（對方沒回應、回應格式錯、超時），怎麼辦？
   - 選項：重試？跳過？降級？報錯給使用者？
6. **橋接需求**：有沒有兩個系統「講不同語言」需要翻譯器（adapter/bridge）的情況？
7. **新增連線影響**：加了這條新連線之後，會不會影響已有的其他連線？

**快速版提問（2 題）**：
1. 這個功能需要跟哪些東西溝通？用什麼方式？
2. 溝通失敗的時候怎麼辦？

### 📦 藍圖三：分工地圖（程式碼怎麼組織？）

**對應工程概念**：模組化、關注點分離、套件結構、耦合度
**建築比喻**：哪個工班負責哪面牆？拆了一面牆會不會影響隔壁？

**完整版提問（5 題）**：

1. **放在哪裡**：這段新程式碼應該放在現有的哪個資料夾/模組裡？還是需要建新的？
2. **隔離程度**：如果三個月後要把這塊整個換掉，需要動到幾個檔案？（理想是只動自己的）
3. **共用部件**：有沒有什麼邏輯是「已經有人寫過」的？要共用還是複製一份？
4. **命名約定**：新建的檔案/函式/變數，有沒有跟現有的命名風格一致？
5. **改動影響範圍**：改了這裡之後，有沒有可能「意外」影響到不相關的地方？

**快速版提問（2 題）**：
1. 新程式碼放在現有結構的哪裡？
2. 改了這裡，還有哪裡可能被影響？

### 🖥️ 藍圖四：部署地圖（實際跑起來要什麼？）

**對應工程概念**：部署架構、環境配置、啟動順序、資源需求
**建築比喻**：承重牆在哪？地基能承受嗎？幾樓放什麼設備？

**完整版提問（5 題）**：

1. **執行環境**：這個功能跑在哪裡？本地 Mac Mini？雲端？Docker 容器？
2. **環境依賴**：需要什麼套件/工具/API Key/環境變數才能跑？
3. **啟動順序**：有沒有「A 必須先啟動 B 才能跑」的依賴？
4. **資源消耗**：跑起來吃多少記憶體/CPU/磁碟？會不會跟現有服務搶資源？
5. **本地 vs 部署差異**：在你電腦上能跑的東西，搬到正式環境有沒有什麼不一樣？

**快速版提問（1 題）**：
1. 跑這個需要什麼環境？有沒有什麼特殊依賴？

### 🎬 藍圖五：走一遍（端到端驗證）

**對應工程概念**：場景視圖、使用者故事、E2E 測試、BDD 行為驅動開發
**建築比喻**：人從大門走進來，一路到每個房間再出去，路通不通？

> 這張圖是前面四張的「驗證層」——你可能每張圖都畫了，但組合起來走不通。

**完整版提問（5 題）**：

1. **觸發點**：使用者做了什麼動作（或什麼事件發生）來啟動這個功能？
2. **快樂路徑**：一切順利的話，從觸發到拿到結果，中間經過幾個步驟？一步步走一遍。
3. **岔路**：哪一步可能出現「不是預期的情況」？每個岔路怎麼處理？
4. **可觀察性**：執行的過程中，你怎麼知道「它正在跑」還是「它卡住了」？有 log 嗎？
5. **成功定義**：怎麼確認它真的做對了？（不是「看起來對」，是「確認對」）

**快速版提問（2 題）**：
1. 從觸發到結果，走一遍每個步驟。
2. 怎麼確認它真的做對了？

## 任務規模分級

| 規模 | 判斷條件 | 藍圖深度 | 估計提問數 |
|------|---------|---------|-----------|
| **S 小改動** | 單一檔案修改、bug 修復、設定調整 | 不需要 preflight（直接做） | 0 |
| **M 中型任務** | 新功能但不涉及多系統、模組內調整 | 快速版（每張圖 1-2 題） | 5-8 題 |
| **L 大型任務** | 新模組、多系統整合、重構、架構變更 | 標準版（每張圖 3-5 題） | 15-20 題 |
| **XL 架構級** | 全新子系統、跨系統遷移、核心邏輯重寫 | 完整版（五張圖全展開 + 歷史教訓比對） | 25-30 題 |

**自動判斷邏輯**：
- 提到「接起來」「整合」「多個系統」→ 至少 L
- 提到「新建」「從零開始」→ 至少 L
- 提到「重構」「遷移」「架構」→ XL
- 提到「改一下」「加一個」「修一個」→ M 或 S
- 不確定時，預設 M，跑完快速版後再判斷是否需要升級

## 歷史教訓載入機制

### 與 dev-retro 的銜接

每次 preflight 啟動時，檢查是否有相關的歷史教訓：

1. **情境比對**：從 dev-retro 的結晶紀錄中，用任務描述做關鍵字比對
2. **主動提示**：如果找到相關紀錄，主動在對應藍圖的提問前插入提醒：
   ```
   ⚠️ 歷史教訓：上次做類似的整合任務時，踩過「訊息格式不一致」的坑。
   根因：沒有在開發前約定介面契約。
   建議：這次特別注意連線地圖的第 2、6 題。
   ```
3. **新增檢查項**：如果 retro 的 L5 規則產出了新的防線規則，自動併入對應藍圖的提問中

### 記憶結構

歷史教訓以 knowledge-lattice 的 Crystal 格式儲存：
```
Crystal Type: dev-lesson
觸發情境: [描述]
缺失藍圖: [功能/連線/分工/部署/動線]
根因標籤: [介面契約/狀態管理/錯誤處理/耦合/環境差異/...]
防線規則: [新增到 preflight 的具體檢查項]
```

2. 存到專案目錄作為開發紀錄
3. 開發完成後交給 dev-retro 做回溯比對

## 護欄

### 硬閘

**HG-PF-NO-SKIP-MAP-2**：連線地圖（藍圖二）在 M 以上的任務中不可跳過。這是非工程師最常踩坑的藍圖。即使使用者說「這個很簡單不需要」，至少要問快速版 2 題。

**HG-PF-NO-JARGON**：所有提問必須用白話。如果偵測到自己在用工程術語（API、interface、coupling、middleware、adapter），必須立刻附上白話翻譯或建築比喻。

**HG-PF-HISTORY-CHECK**：如果 dev-retro 有相關歷史教訓且根因標籤匹配當前任務，必須主動提示。不可因為「之前講過了」就省略。

### 軟閘

**SG-PF-OVERASK**：避免過度提問。S 級任務不需要 preflight。M 級任務快速版就夠。不要用 XL 的陣仗處理 M 的任務。

**SG-PF-PACE**：每次最多問 3-5 題，等使用者回答後再推進下一張藍圖。不要一次把 30 題全丟出來。

**SG-PF-CONTEXT-AWARE**：如果使用者在對話中已經自然回答了某張藍圖的問題，不需要重複問。標記「已從上下文推斷」即可。

## DNA27 親和對照

啟用 dev-preflight 時：
- Persona 旋鈕建議：tone → WARM、pace → STEADY、initiative → ASK
  - 這是提問導向的 Skill，不是主導推進的。耐心引導，不催促。
- 偏好觸發的反射叢集：Group A（認知層——問題品質判斷）、Group C（結構化思考）
- 限制使用的反射叢集：Group E（不展開商業分析或戰略推演，那不是 preflight 的職責）

與其他外掛的協同：
- **dev-retro**：閉環夥伴。retro 的 L5 防線規則回灌 preflight 的檢查項；preflight 的藍圖摘要卡是 retro 回溯的比對基準。
- **plan-engine**：plan-engine 的 Research 階段可調用 preflight 做認知補全。preflight 的藍圖摘要卡可作為 plan.md 的輸入。
- **qa-auditor**：preflight 管「寫之前」，qa-auditor 管「寫完之後」。兩者覆蓋開發生命週期的前後端。
- **dse**：dse 管「技術融合可行性研究」（深度），preflight 管「日常開發的系統性遺漏預防」（廣度）。
