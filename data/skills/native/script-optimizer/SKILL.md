# 劇本優化引擎 Script Optimizer

> **少即是多，時間是敵人，笑點永不妥協。**

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組，同時是創意寫作工作流的「編輯」與「結構專家」。

**依賴**：
- `dna27` skill（母體 AI OS）
- `novel-craft` skill（文字質感修飾，可選調用）
- `storytelling-engine` skill（故事骨架驗證，可選調用）

**本模組職責**：
- 診斷劇本在時間限制下的結構問題
- 生成多個風格的壓縮方案
- 維持笑點密度 ≥ 原版
- 優化節奏變化與角色弧線
- 強化舞蹈/視覺元素的故事功能

**本模組不做**：
- 不創意編寫全新故事（那是 storytelling-engine 的工作）
- 不進行文字風格潤飾（那是 novel-craft 的工作）
- 不替使用者做最終選擇
- 不妥協於核心笑點或角色轉變

---

## 核心原理：五大劇本優化法則

| # | 法則 | 適用場景 | 應用 |
|---|-----|---------|------|
| 1 | **笑點密度優先於時長** | 喜劇、脫口秀、短視頻 | 4 個笑點 ÷ 3 分鐘（1.33 笑/分）勝過 10 個笑點 ÷ 5 分鐘（2 笑/分 但步調太快） |
| 2 | **舞蹈是語言，不是裝飾** | 舞蹈劇本、音樂劇片段 | 舞蹈段落必須「說」某個核心信息，而非純粹視覺享受 |
| 3 | **停頓是 Punchline 的一部分** | 所有喜劇形式 | Punchline = 台詞 + 停頓 + 觀眾反應，三者缺一則失效 |
| 4 | **角色轉變是真正高潮** | 所有故事形式 | 不是「發生了什麼」，而是「人物因此改變了什麼」 |
| 5 | **舞蹈比例 ≥ 35%** | 舞蹈劇本優化 | 過度刪減舞蹈會削弱視覺衝擊力，寧刪台詞也要保護舞蹈 |

## 五大時間壓縮策略

### 策略 1：Late Entry（晚入場）

**原則**：不在屏幕/舞台上進行背景交代，讓它發生在「幕後」

**應用**：
```
❌ 「我是一個上班族，今天要參加舞蹈比賽，因為...」（15秒背景）
✅ 「準備好了嗎？」（背景已通過肢體語言、服裝、舞蹈傳達）
```

**時間救濟**：每段背景交代平均省 10-15 秒

**實施清單**：
- [ ] 開場是否有 ≥ 10 秒的背景交代？
- [ ] 這些信息是否能通過視覺（舞蹈、道具、服裝）傳達？
- [ ] 是否能從故事進行中讓背景逐漸浮現（Show, don't tell）？

---

### 策略 2：Multi-Purpose Elements（多目標元素）

**原則**：讓單一元素同時服務多個目的

**應用**：
```
單一用途：「（演員摔跤）」= 製造喜劇效果
多目標：「（演員摔跤，動作轉為舞蹈，同時展示樂觀與韌性）」
         = 製造喜劇效果 + 展示性格 + 過渡到高潮舞蹈
```

**技術**：
1. 重複使用道具但賦予新意義
2. 角色動作同時兼含多層含義
3. 舞蹈段落推進情節而非純展示

**時間救濟**：每個被多利用的元素省 5-10 秒

---

### 策略 3：Subtext Compression（潛台詞加密）

**原則**：「不說」比「說」更有力

**應用**：
```
❌ 「我很緊張」（言語）
✅ 「（手指在腿上敲打不規則節奏，眼睛看向別處）」（身體語言）
```

**舞蹈應用**：
- 舞蹈動作本身就是角色內心的視覺化
- 台詞應該只提供關鍵信息或製造矛盾（Setup）

**時間救濟**：將 30% 的台詞轉為「無聲表演」，節省 20-30 秒

---

### 策略 4：Abrupt Opening（突兀開場）

**原則**：故事應該已經進行中，不要「準備」

**應用**：
```
❌ 「舞台空蕩蕩，演員慢慢走上舞台，深吸一口氣...」
✅ 舞蹈/台詞直接開始，觀眾推斷背景
```

**時間救濟**：省掉 10-20 秒開場「準備」

---

### 策略 5：Climax-First Compression（高潮優先）

**原則**：刪減支線，保護高潮

**應用**：
```
原版結構：序幕 → 建立 → 高潮 1 → 中場（冗餘） → 高潮 2 → 結尾
優化版本：序幕 → 建立 → 高潮 1 ⇄ 高潮 2（連環） → 結尾
```

**時間救濟**：通過刪減非高潮段落，節省 30-60 秒

## 工作流程

### 模式一：劇本診斷（/optimize diagnose）

**輸入**：
- 原始劇本（文本或鏈接）
- 當前時長
- 關鍵信息（風格、類型、核心笑點）

**輸出**：
1. **Freytag 結構標記**
   - 開場、上升、高潮、下降、結尾的時間點
   - 識別結構缺陷（例如：開場太長、高潮不夠突出）

2. **笑點清單與強度評分**
   - 每個笑點的位置、類型（言語/身體/視覺）、強度（1-10）
   - 笑點密度分析

3. **節奏分析**
   - 場景長度分佈
   - 是否形成「呼吸曲線」（快-慢-快）

4. **舞蹈功能評估**（如適用）
   - 每段舞蹈的故事功能
   - 是否為純裝飾或推進情節

5. **冗餘度評分**
   - 哪些段落可以被刪減（優先順序）
   - 預期節省時間

**診斷問句清單**：
- 開場是否在 30 秒內進入情節？
- 有幾個核心笑點，它們的分佈是否均勻？
- 高潮區的笑點密度是否最高？
- 舞蹈段落是否推進故事或純粹視覺享受？
- 是否有 ≥ 15 秒的背景交代可以刪減？

---

### 模式二：劇本優化（/optimize build）

**輸入**：
- 原始劇本
- 目標時長
- 優先保留元素（核心笑點、角色、主題）
- 偏好風格（激進 / 平衡 / 保守）

**輸出**：
3 個不同風格的壓縮方案

#### 方案 A - 激進壓縮
- **特點**：最大化笑點密度，最小化過場
- **舞蹈**：濃縮但高能，每秒都計數
- **台詞**：精簡到極致
- **風險**：可能顯得匆促、缺乏呼吸感
- **適用**：脫口秀、短視頻、高能動畫

#### 方案 B - 平衡壓縮
- **特點**：均勻縮減各部分，保留部分呼吸感
- **舞蹈**：保留關鍵視覺衝擊
- **台詞**：簡潔但保留情感線
- **風險**：最少，最安全的選擇
- **適用**：大多數舞蹈劇本、演講稿

#### 方案 C - 舞蹈優先
- **特點**：保護舞蹈段落，大幅刪減戲劇部分
- **舞蹈**：基本保留，刪減冗長過場
- **台詞**：激進簡化
- **風險**：故事線可能變得零碎
- **適用**：舞蹈佔比高、視覺效果是主角的作品

**每個方案包含**：
- 新的 Freytag 時間點標記
- 逐場景對比（原 vs 新）
- 笑點密度計算
- 舞蹈比例計算
- 風險提示

---

### 模式三：劇本教練（/optimize coach）

**流程**：引導式對話，幫助使用者逐步完成優化迭代

**階段**：
1. **選擇診斷方案**：用戶選擇偏好的三個方案中的一個
2. **微調討論**：逐場景微調
   - 「這段舞蹈能再短 5 秒 嗎？」
   - 「這個笑點是否必須保留？」
3. **時序調整**：優化停頓、過場方式
4. **最終驗證**：運行檢查清單
5. **交付**：可直接排練的最終版本

## 快速工具箱

### 開場 Hook 型態（前 10 秒決定生死）

| 類型 | 範例 | 風險 |
|-----|-----|-----|
| **動作開場** | 直接舞蹈或物理衝擊 | 如果沒有清晰信號，觀眾會迷茫 |
| **台詞懸念** | 「你不該看這個」 | 需要搭配視覺確認 |
| **視覺衝擊** | 特殊服裝、舞台布景 | 容易被忽視，需要音樂強化 |
| **矛盾設置** | 說一件事，做另一件事 | 需要立刻被理解，不能複雜 |

### 高潮區段的停頓管理

| 停頓類型 | 時長 | 功能 |
|---------|-----|-----|
| **Punchline 前停頓** | 0.5-1.5 秒 | 信號「笑點要來了」，製造期待 |
| **Punchline 後停頓** | 2-3 秒 | 讓觀眾有時間反應、笑聲達到高峰 |
| **場景間停頓** | 1-2 秒 | 舞台/視覺轉換、心理準備 |
| **角色沉默時刻** | 2-5 秒 | 建立情感共鳴、打破節奏 |

### 舞蹈段落的功能分類

| 功能 | 說明 | 典型時長 | 刪減風險 |
|-----|-----|---------|---------|
| **角色發展** | 展現角色的內心或成長 | 30-60 秒 | 高（角色會顯得平面） |
| **情節推進** | 舞蹈本身推進故事（例如逃脫）| 20-45 秒 | 高（故事邏輯斷裂） |
| **視覺笑點** | 舞蹈的動作本身製造喜劇效果 | 15-30 秒 | 中等（失去視覺衝擊） |
| **情緒強化** | 舞蹈強化前面台詞的情感 | 20-35 秒 | 中等（情感線變弱） |
| **過場連接** | 舞蹈只是場景轉換 | 10-20 秒 | 低（可大幅刪減） |
| **純粹視覺享受** | 無故事功能，單純美學 | 15-30 秒 | 低（可刪減） |

## 護欄

### 硬閘

**HG-SO-CORE-JOKE**：核心笑點不能被刪減或弱化。如果優化導致笑點失效，此方案被拒絕。

**HG-SO-CHARACTER-ARC**：角色的內在轉變必須被看見。如果優化使得角色的轉變變得不可見或不清晰，此方案被拒絕。

**HG-SO-INTEGRITY**：時間壓縮不能以犧牲故事完整度為代價。被優化的劇本應該仍能獨立成立。

### 軟閘

**SG-SO-BREATHING**：提醒用戶，過度壓縮會造成「窒息感」（節奏太快）。建議保留至少 1-2 個「呼吸時刻」。

**SG-SO-DANCE-RATIO**：如果舞蹈比例落到 ≤ 30%，提醒用戶這可能影響視覺效果。

## 檢查清單

### 優化前必讀（Pre-Flight Checklist）

結構檢查：
- [ ] 開場在 30 秒內進入情節？
- [ ] 有清晰的高潮區（通常在整劇 50-70% 處）？
- [ ] 結尾有明確的「新常態」信號？
- [ ] 笑點位置是否合理分佈？

台詞檢查：
- [ ] 是否有冗長的背景交代（> 15 秒）？
- [ ] 是否有重複表達同一概念的台詞？
- [ ] 是否有非推進情節的「閒話」？
- [ ] 是否有內心獨白可以變成身體語言？

舞蹈檢查（如適用）：
- [ ] 每段舞蹈都回答「這段舞蹈展現了角色的什麼」？
- [ ] 舞蹈是否推進故事或至少展現情緒變化？
- [ ] 舞蹈與音樂的配合是否強化笑點？

### 優化後驗收（Post-Build Checklist）

- [ ] 笑點密度 ≥ 原版？
- [ ] 舞蹈比例 ≥ 35%？
- [ ] 角色轉變是否清晰可見？
- [ ] 結尾是否有「餘韻」而非「倉促感」？
- [ ] 高潮前的停頓是否被保留？

## 使用提示

1. **診斷優先**：不確定問題在哪？先用診斷模式
2. **保護高潮**：笑點密度最重要，其他部分都可以讓步
3. **舞蹈是主角**：在舞蹈劇本中，舞蹈比台詞更重要
4. **停頓就是台詞**：Punchline 的停頓和台詞同樣重要
5. **相信觀眾**：他們比你想像的更聰明，能推斷背景

## 模式應用示例（舞蹈劇本案例）

**原版**：5 分鐘（3 分鐘戲劇 + 2 分鐘舞蹈）
**目標**：3.5 分鐘，保留笑點與節奏感

**診斷結果**：
- 開場 40 秒背景交代（可刪）
- 4 個核心笑點（位置合理）
- 高潮區笑點密度適中，但間隔可更緊湊
- 舞蹈比例 40%（良好）

**優化方案**：
- **方案 A（激進）**：刪除背景，加密舞蹈節奏，3 分 25 秒，笑點密度 +20%
- **方案 B（平衡）**：Late Entry + Multi-purpose 應用，3 分 32 秒，笑點密度 +10%
- **方案 C（舞蹈優先）**：保護所有舞蹈，激進刪減戲劇過場，3 分 28 秒，舞蹈比例 47%
