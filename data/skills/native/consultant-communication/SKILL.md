# Consultant Communication Engine：顧問式溝通引擎

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：MUSEON-DNA27-vNext（提供 Kernel 護欄、迴圈路由、模式切換、Persona 旋鈕、演化引擎、記憶系統）

**本模組職責**：
- 定義顧問式溝通的五大技術棧（金字塔/MECE/SCQA/HBR 8S/個案方法）
- 提供結構化思維與表達的操作指南
- 提供個案研究的撰寫與分析框架

**本模組不做**：
- 不重複定義迴圈（fast/exploration/slow）、模式（civil/evolution）、Persona 旋鈕

- 不支援欺騙性或操控性溝通

## 技術一：金字塔原則（Pyramid Principle）

### 核心概念

金字塔原則是一種「結論先行」的溝通結構，由麥肯錫顧問 Barbara Minto 在 1970 年代開發。

**核心邏輯**：
- 思考時由下而上（資料→分析→結論）
- 表達時由上而下（結論→論點→證據）

**三層結構**：
```
        [主要訊息/結論]           ← 頂層：一句話回答「所以呢？」
       /      |      \
   [論點1]  [論點2]  [論點3]      ← 中層：支撐結論的 2-4 個關鍵論點
   / | \    / | \    / | \
 [證據] [證據] [證據] ...         ← 底層：支撐論點的數據/事實
```

### 垂直關係規則

每一層都是下一層的摘要。測試方法：
- 上層應該能回答「所以呢？」
- 下層應該能回答「為什麼？」或「怎麼做？」

### 水平關係規則

同一層的元素必須：
1. **邏輯同類**：都是原因、都是步驟、都是問題…
2. **邏輯排序**：時間順序、重要性順序、結構順序
3. **MECE**：相互獨立、完全窮盡

### 應用場景

| 場景 | 頂層 | 中層 | 底層 |
|------|------|------|------|
| 決策提案 | 建議行動 | 2-3 個理由 | 支撐數據 |
| 問題診斷 | 根本原因 | 原因分類 | 具體症狀 |
| 狀態匯報 | 結論/現況 | 關鍵發現 | 詳細數據 |
| 專案規劃 | 目標 | 關鍵里程碑 | 具體任務 |

詳見 `references/pyramid-principle.md`

## 技術二：MECE 原則（Mutually Exclusive, Collectively Exhaustive）

### 核心概念

MECE（發音 me-see）是一種分類原則，確保：
- **ME（相互獨立）**：每個類別不重疊，一個元素只能屬於一個類別
- **CE（完全窮盡）**：所有類別加起來涵蓋所有可能性，沒有遺漏

### MECE vs 非 MECE 範例

| 分類方式 | MECE？ | 原因 |
|----------|--------|------|
| 男性 / 女性 / 非二元 | ✅ | 不重疊、涵蓋所有 |
| 年輕人 / 老年人 | ❌ | 中年人漏掉了（不窮盡） |
| 線上客戶 / VIP 客戶 | ❌ | VIP 可能也是線上的（重疊） |
| 收入 / 成本 | ✅ | 利潤拆解的經典 MECE |
| 內部因素 / 外部因素 | ✅ | 問題歸因的經典 MECE |

### 五種建構 MECE 的策略

1. **X / Not X 法**：最簡單的二分法（可控 vs 不可控、內部 vs 外部）
2. **流程法**：按時間或流程階段切分（獲客→轉換→留存→推薦）
3. **要素法**：按組成要素切分（人/財/物/資訊）
4. **矩陣法**：兩個維度交叉（重要性 × 緊急性）
5. **公式法**：用數學公式拆解（利潤 = 收入 - 成本）

### 常見 MECE 框架速查

| 框架 | 結構 | 適用場景 |
|------|------|----------|
| 3C | Customer / Company / Competitor | 市場分析 |
| 4P | Product / Price / Place / Promotion | 行銷策略 |
| SWOT | Strengths / Weaknesses / Opportunities / Threats | 策略評估 |
| 利潤拆解 | 收入（價 × 量）/ 成本（固定 / 變動） | 財務診斷 |
| 價值鏈 | 研發→採購→生產→行銷→銷售→服務 | 營運分析 |

詳見 `references/mece-framework.md`

## 技術三：SCQA 框架（Situation-Complication-Question-Answer）

### 核心概念

SCQA 是一種敘事框架，用於為你的「答案」提供脈絡，讓聽眾知道「為什麼要聽」。

**四個元素**：

| 元素 | 功能 | 範例 |
|------|------|------|
| **S**ituation（情境） | 建立共識的起點 | 「我們的市場份額連續三年穩定成長…」 |
| **C**omplication（挑戰） | 引入張力與問題 | 「但是新進競爭者以破壞性價格進入…」 |
| **Q**uestion（問題） | 明確要回答什麼 | 「我們應該如何因應？」 |
| **A**nswer（答案） | 你的核心建議 | 「建議採取差異化策略，聚焦高端市場」 |

### SCQA 變體

| 變體 | 結構 | 適用場景 |
|------|------|----------|
| **標準 SCQA** | S→C→Q→A | 大多數商業簡報 |
| **直接式** | A→S→C→Q→A（展開） | 高管已知背景，想快速看結論 |
| **懸念式** | S→C→C→C→Q→A | 需要建立急迫感 |
| **故事式** | S→C→嘗試→失敗→Q→A | 需要情感連結 |

### SCQA + 金字塔 整合

```
[SCQA 開場]
  └─ S: 情境
  └─ C: 挑戰  
  └─ Q: 問題
  └─ A: 核心建議（金字塔頂層）
        └─ [論點1] + 證據
        └─ [論點2] + 證據
        └─ [論點3] + 證據
[結尾：行動呼籲]
```

## 技術四：HBR 8S 寫作原則

### 核心概念

基於神經科學研究，Bill Birchard 歸納出八個能觸發大腦愉悅反應的寫作特徵。

### 八大 S 原則

| 原則 | 英文 | 操作指引 | 反面教材 |
|------|------|----------|----------|
| **簡單** | Simplicity | 用白話說複雜概念，一句一主旨 | 堆砌術語、長句套長句 |
| **具體** | Specificity | 用數字、名字、細節取代抽象詞 | 「顯著提升」→「提升 23%」 |
| **驚喜** | Surprise | 打破預期、挑戰常識 | 陳腔濫調、意料之中 |
| **動人** | Stirring | 有畫面感、感官語言 | 純概念、無感情 |
| **吸引** | Seductiveness | 製造好奇、留懸念 | 一次說完、無懸念 |
| **聰明** | Smart ideas | 有洞見而非常識 | 人人都知道的道理 |
| **社交** | Social | 連結人際、用「你」「我們」 | 純第三人稱、冷冰冰 |
| **故事** | Storytelling | 用敘事包裝論點 | 純數據羅列 |

### 8S 檢核表

寫完一篇文章後，逐項檢查：
- [ ] 是否用最簡單的方式說清楚了？
- [ ] 是否有足夠的具體數字和細節？
- [ ] 是否有至少一個出人意料的觀點？
- [ ] 是否有畫面感、讀者能「看見」？
- [ ] 開頭是否讓人想繼續讀下去？
- [ ] 是否提供了讀者不知道的洞見？
- [ ] 是否與讀者建立了連結？
- [ ] 是否用故事或案例支撐論點？

## 技術五：個案研究方法（Case Study Method）

### 核心概念

個案研究是一種「用故事教複雜概念」的方法，源自哈佛商學院。與一般成功故事不同，教學個案：
- 從決策者視角撰寫
- 資訊刻意不完整
- 結局是開放式困境
- 目的是引發思考而非提供答案

### 個案的五大核心元素

#### 1. 主角（Protagonist）

主角是故事的核心決策者，設計要點：
- **有決策權**：能影響結果的人
- **面臨困境**：有需要分析和行動的挑戰
- **可共感**：讀者能理解並投入

#### 2. 鉤子（Hook）

將各部分串聯起來的統攝性議題：
- 不是立即明顯的
- 需要讀者自己「發現」
- 連結到特定理論或概念

#### 3. 開場（Opening）

最重要的部分，功能：
- 舞台設定
- 抓住注意力
- 建立急迫感

#### 4. 張力與衝突

驅動故事的力量：
- 引發興趣的議題
- 提供不確定性
- 沒有明顯「正確答案」

#### 5. 開放式結尾

留下決策點：
- 不給標準答案
- 邀請讀者做出判斷
- 反映真實世界的模糊性

### 個案結構：三幕式

**Part I — 背景與挑戰**
- 組織背景
- 主角介紹
- 大挑戰的組成部分
- 其他重要角色

**Part II — 過程：策略、收穫、挫折**
- 最初的嘗試
- 如何克服障礙
- 關鍵決策時刻
- 變革推動者 vs 阻力

**Part III — 前路：剩餘挑戰**
- 還沒解決的問題
- 開放式決策點
- 討論問題

### A-B-T 敘事模板

極簡但強大的三句話結構：

| 元素 | 功能 | 連接詞 |
|------|------|--------|
| **A**nd（而且） | 建立常態 | 「…而且…」 |
| **B**ut（但是） | 引入衝突 | 「但是…」 |
| **T**herefore（因此） | 推向決策 | 「因此…」 |

**範例**：
「公司連續三年成長 20%，**而且**市場份額穩定擴大。**但是**新進競爭者以破壞性價格進入市場。**因此**執行長必須決定是否降價應戰或堅持高端定位。」

詳見 `references/case-study-engine.md`

## 工作流程

### Step 1：釐清溝通目的

先回答這些問題：
- 聽眾是誰？他們已經知道什麼？
- 你要回答什麼問題？
- 聽眾需要做什麼決定或行動？
- 你有多少時間/篇幅？

### Step 2：選擇技術組合

| 溝通目的 | 建議技術組合 |
|----------|--------------|
| 高管決策匯報 | 金字塔 + SCQA（直接式） |
| 專案簡報 | SCQA + 金字塔 |
| 商業文章 | 8S + SCQA |
| 問題分析 | MECE + 金字塔 |
| 經驗萃取 | 個案方法 |
| 教育訓練 | 個案方法 + 討論問題 |
| 電梯簡報 | A-B-T |

### Step 3：結構化內容

根據選定的技術組合，用對應的模板組織內容。

### Step 4：審計與優化

用審計工具檢查，逐項改進。

### Step 5：練習與迭代

溝通是技能，需要刻意練習：
- 每次簡報後自我複盤
- 蒐集聽眾反饋
- 建立個人案例庫
