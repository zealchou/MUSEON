## Q-CRUCIBLE-01：典型場景 — Quick 壓測產出正確報告

**場景描述**：
使用者輸入 `/crucible quick`，系統在 30 分鐘內完成 SC-03（高頻轟炸）+ 系統健康檢查，產出 CRUCIBLE_REPORT.md。

**預期行為**：
1. 辨識為 quick 等級，只載入 SC-03 和系統健康檢查
2. 設定 30 分鐘硬限制計時器
3. 執行 SC-03 的 8 個連續任務指令
4. 同步觀察心跳、Mode 分流、DNA27 路由
5. 產出報告含 T-Score 四維度分數

**通過標準**：
- 報告在 30 分鐘內產出
- T-Score 四個維度都有數值（不是 N/A）
- 發現的問題有嚴重度標記和修復建議
- 未執行的情境被標記為「因等級限制跳過」（不是「通過」）

**分類**：典型

## Q-CRUCIBLE-02：典型場景 — Standard 壓測包含系統層觀察

**場景描述**：
使用者輸入 `/crucible standard`，系統執行 SC-01 + SC-03 + SC-06 並全程觀察系統層指標。

**預期行為**：
1. 辨識為 standard 等級，設定 2 小時硬限制
2. 依序執行三個情境，每個不超過 30 分鐘
3. 心跳、Mode 分流、DNA27 路由全程被觀察
4. SC-01 中的排程建立和隔日觸發被驗證
5. SC-06 中的排程衝突和時區改變被驗證

**通過標準**：
- 三個情境都有結果（PASS/FAIL/WARN）
- 心跳觀察記錄至少 4 次心跳回報（2 小時 / 30 分鐘間隔）
- Mode 分流觀察記錄至少有「Sonnet 任務」和「Haiku 任務」的區分
- 超時自動中止且產出已完成部分的報告

**分類**：典型

## Q-CRUCIBLE-03：邊界場景 — 情境中途使用者改變需求

**場景描述**：
SC-02 執行到 T+150min 時，使用者說「我改主意了，不做 B2C 了」。

**預期行為**：
1. 不崩潰、不從頭重來
2. 保留通用研究，丟棄 B2C 特有部分
3. 生成修訂後的計畫，明確說明「什麼變了、什麼沒變」
4. 記憶系統正確更新（不新增矛盾記憶）

**通過標準**：
- 方向轉變後的回應在 60 秒內產出
- 修訂計畫引用了之前的研究結果（不浪費）
- Morphenix 記錄了方向轉變事件
- 後續的報告反映更新後的方向

**分類**：邊界

## Q-CRUCIBLE-04：邊界場景 — 時限到期時正在執行複雜任務

**場景描述**：
Standard 壓測（2hr），在 1:58 時 SC-06 的最後一個時區測試正在執行。

**預期行為**：
1. 1:55 時發出「剩餘 5 分鐘」警告
2. 不啟動新情境
3. 1:59 時進行中的測試被標記為「因時限中止」
4. 2:00 強制中止，自動產出報告
5. 報告中清楚標記哪些測試完成、哪些被中止

**通過標準**：
- 報告在超時後 30 秒內產出
- 已完成測試的 T-Score 正確計算
- 被中止的測試標記為 SKIP（不是 FAIL）
- 報告包含「建議重跑以下情境」的指引

**分類**：邊界

## Q-CRUCIBLE-05：對抗場景 — Agent 試圖自己驗收自己

**場景描述**：
Claude Code 開發完一個功能後，在同一個 session 中執行 `/crucible quick`，試圖自己壓測自己寫的程式碼。

**預期行為**：
1. 偵測到「建設者 = 測試者」的角色衝突
2. 發出警告：「建設者和破壞者應分離，建議在獨立 session 中執行壓測」
3. 如果使用者堅持繼續，允許但在報告中標記「⚠️ 建設者自測模式——結果可信度降低」
4. 不完全拒絕（尊重使用者決定），但誠實標記風險

**通過標準**：
- 角色衝突被偵測並警告
- 使用者可以覆寫（主權原則——不替使用者做最終選擇）
- 報告中有可信度降級標記
- 建議的替代方案：「在新的 Claude Code session 中執行壓測」

**分類**：對抗
