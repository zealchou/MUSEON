# Env-Radar：環境雷達 — 外部環境掃描與訊號情報引擎

> **閉著眼睛進化，只能適應昨天的世界。**
> meta-learning 管「怎麼學」，Env-Radar 管「該學什麼」——方向感。

## 外掛合約

此 Skill 為 DNA27 核心的外掛模組（pluggable plus）。

**依賴**：`dna27` skill（母體 AI OS，提供 Kernel 護欄、迴圈路由、模式切換、Persona 旋鈕、演化引擎、六層記憶系統。本模組不可脫離 dna27 獨立運作。）

**本模組職責**：
- 系統性掃描 AI 產業的外部環境變化（技術、市場、競品、政策、用戶行為）
- 將原始外部訊號分類、評估、轉譯為 MUSEON 可操作的「演化壓力」
- 維護技術雷達（Technology Radar），追蹤技術成熟度與採用時機
- 偵測弱訊號（Weak Signals），在趨勢成形前提早捕捉
- 產出演化壓力報告，餵入 morphenix 的迭代方向決策
- 為 master-strategy 和 gap 提供環境情報基礎

**本模組不做**：
- 不做 AI Agent 市場的商品缺口分析（那是 gap 的工作）
- 不做戰略判斷與路徑選擇（那是 master-strategy 的工作）
- 不做系統自身的品質度量（那是 eval-engine 的工作）
- 不做系統的自主迭代執行（那是 morphenix 的工作）
- 不做 AI 技術的深度融合驗證（那是 dse 的工作）

**與其他外掛的關係**：

| 外掛 | 協作方式 |
|-----|---------|
| morphenix | Env-Radar 提供「該往哪裡進化」的環境壓力，morphenix 執行迭代 |
| eval-engine | eval-engine 提供內部品質數據，Env-Radar 提供外部品質標竿，兩者比對發現差距 |
| gap | gap 專攻 AI Agent 市場缺口（微觀），Env-Radar 監控整體環境（宏觀）|
| master-strategy | master-strategy 做戰略判斷時，Env-Radar 提供環境情報作為判斷依據 |
| dse | dse 做技術融合驗證時，Env-Radar 的技術雷達提供「該驗證哪些技術」的方向 |
| knowledge-lattice | 環境掃描中的洞見可結晶為 Hypothesis/Insight 結晶 |
| meta-learning | meta-learning 管學習方法，Env-Radar 管學習方向——兩者組合決定「用什麼方法學什麼東西」|

## 核心架構

### 四層掃描模型（Scan Stack）

```
L4 演化壓力層 ──── 「所以 MUSEON 該怎麼動？」
     ↑ 壓力轉譯
L3 洞見層 ──────── 「這些變化意味著什麼？」
     ↑ 模式辨識
L2 分析層 ──────── 「這些訊號的關聯是什麼？」
     ↑ 交叉比對
L1 訊號層 ──────── 「外面發生了什麼？」
     ↑ 原始資料
[ 外部環境 ] ────── AI 產業、技術、市場、政策、用戶
```

### 六維環境模型（PESTEL 改良版）

針對 MUSEON 的 AI 顧問業務場景，將經典 PESTEL 框架適配為：

| 維度 | 代號 | 掃描範圍 | 範例 |
|------|------|---------|------|
| **技術趨勢** | T | AI 模型進化、新架構、新工具鏈 | Claude 新版本、MCP 協議更新、新 Agent 框架 |
| **市場動態** | M | AI Agent 市場、Skill 商品市場、SME AI 採用 | Claude Skills 上架趨勢、GPTs 熱門品類 |
| **競品情報** | C | 直接競品 + 鄰近競品 + 替代品 | 其他 AI 顧問、自動化平台、DIY 解決方案 |
| **用戶行為** | U | 台灣 SME 對 AI 的認知/需求/採用變化 | AI 焦慮→好奇→嘗試→依賴 的階段遷移 |
| **政策法規** | P | AI 法規、數據隱私、產業政策 | 台灣 AI 基本法、GDPR 影響、產業補助 |
| **生態系統** | E | AI 產業鏈、合作夥伴、平台演化 | Anthropic 平台政策、API 定價變化、MCP 生態 |

### 訊號分級系統

每個捕捉到的外部訊號，依以下標準分級：

| 等級 | 名稱 | 條件 | MUSEON 動作 |
|------|------|------|-----------|
| 🔴 S1 | 緊急 | 直接影響 MUSEON 核心功能或收入 | 立即通知 + morphenix 緊急迭代 |
| 🟡 S2 | 重要 | 影響競爭力，3 個月內需回應 | 寫入 morphenix 迭代筆記 + 排程 |
| 🟢 S3 | 關注 | 可能影響，需持續追蹤 | 標記追蹤 + 下次掃描重點 |
| ⚪ S4 | 記錄 | 長期可能有影響 | 記入 knowledge-lattice 作 Hypothesis |

**分級判斷標準**：

```
影響分 = 0.4 × 業務衝擊 + 0.3 × 緊急程度 + 0.3 × 不可逆性

業務衝擊：對 MUSEON 營收/功能/品牌的潛在影響 (0-1)
緊急程度：需要回應的時間窗口 (0-1, 越急越高)
不可逆性：錯過機會窗口後能否補救 (0-1, 越不可逆越高)

S1: 影響分 > 0.8
S2: 0.5 < 影響分 ≤ 0.8
S3: 0.3 < 影響分 ≤ 0.5
S4: 影響分 ≤ 0.3
```

## 五大核心功能

### 1. 全景掃描（Panorama Scan）

**觸發**：`/radar` 或 `/scan` 或使用者問「現在市場怎麼樣」

**流程**：
1. 使用 web search 搜尋六維環境的最新動態
2. 每維度收集 3-5 個關鍵訊號
3. 對每個訊號做分級（S1-S4）
4. 交叉比對訊號之間的關聯
5. 綜合產出「環境快照」

**輸出格式**：

```
🛰️ MUSEON 環境雷達 — 全景掃描
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
掃描時間：2026-02-21
掃描範圍：AI Agent 生態 × 台灣 SME 市場

📡 關鍵訊號（按緊急度排序）：

🔴 S1 緊急：
  [T] Anthropic 宣布 Claude Skills 定價調整
      衝擊：直接影響 MUSEON Skill 產品線毛利
      建議：立即評估成本結構，考慮 morphenix 緊急迭代

🟡 S2 重要：
  [C] 台灣某 AI 顧問公司獲 A 輪融資，瞄準 SME 市場
      衝擊：直接競品出現，3 個月內需強化差異化
      建議：啟動 gap 掃描 + ssa-consultant 競爭策略

  [T] MCP 協議 v2.0 發布，支援多 Agent 協作
      衝擊：MUSEON 多 Skill 協作能力需升級
      建議：啟動 dse 技術融合驗證

🟢 S3 關注：
  [U] 台灣中小企業 AI 使用率季度報告：同比 +23%
      含義：市場在膨脹，進入門檻在降低
      追蹤：用戶需求是否從「教育」轉向「實作」

  [P] 數位發展部推出 AI 應用補助計畫
      含義：可能加速 SME 採用，但也可能引入更多競爭者

⚪ S4 記錄：
  [E] Google Gemini 推出 Skill 市場 beta 版
      含義：Agent/Skill 生態可能多平台化，長期需考慮跨平台

📊 環境溫度：
  機會指數：0.72（偏高——市場在膨脹，技術在進步）
  威脅指數：0.45（中等——競品出現但尚未成熟）
  演化壓力：中高——建議 morphenix 在本季至少完成 2 次迭代
```

### 2. 競品追蹤（Competitor Track）

**觸發**：`/competitor` 或使用者問「對手在做什麼」

**追蹤框架（FACT）**：
- **F（Find）**：辨識競品——直接競品、鄰近競品、替代品
- **A（Analyze）**：分析競品的策略、產品、定價、定位
- **C（Confirm）**：交叉驗證情報來源的可靠度
- **T（Tell）**：轉化為 MUSEON 的策略洞見

**競品分類**：

```
直接競品（Same Problem × Same Customer）：
  → 其他 AI 顧問服務商，瞄準台灣 SME

鄰近競品（Same Problem × Different Customer 或 Different Problem × Same Customer）：
  → 大型管顧降維打擊 SME 市場
  → SaaS 自動化工具減少對顧問的需求

替代品（根本不同方案解決相同痛點）：
  → 免費 ChatGPT/Claude 直接用（DIY 路線）
  → 不做數位轉型（維持現狀）
```

**輸出**：競品動態摘要 + 定位差異矩陣 + 策略建議

### 3. 技術雷達（Tech Radar）

**觸發**：`/tech-radar` 或使用者問「現在該關注什麼技術」

**四象限模型**（改良自 ThoughtWorks Technology Radar）：

```
          ┌──────────────────────┐
          │      ADOPT 採用       │  已驗證，應立即融入 MUSEON
          │  （成熟 + 高相關）     │
          ├──────────────────────┤
          │      TRIAL 試驗       │  有潛力，值得在 Sandbox 試跑
          │  （成長 + 高相關）     │
          ├──────────────────────┤
          │     ASSESS 評估       │  值得研究，但暫不行動
          │  （早期 + 中相關）     │
          ├──────────────────────┤
          │      HOLD 觀望        │  知道就好，暫時不碰
          │  （不成熟或低相關）    │
          └──────────────────────┘
```

**技術評估標準**：

| 評估維度 | 權重 | 說明 |
|---------|------|------|
| 技術成熟度 | 0.25 | 是否有穩定的實現？社區規模？文件品質？ |
| MUSEON 相關度 | 0.30 | 對 MUSEON 核心能力或客戶價值的影響程度 |
| 實施難度 | 0.20 | 融入現有架構的成本（時間/技能/風險） |
| 競爭差異化 | 0.25 | 採用後能否產生對手難以複製的優勢 |

**輸出**：技術雷達圖 + 各象限的技術清單 + 優先行動建議

### 4. 弱訊號偵測（Weak Signal Detection）

**觸發**：`/signal` 或自然語言偵測到使用者提到「隱約感覺」「好像」「聽說」

**定義**：弱訊號是尚未形成明確趨勢、但可能預示重大變化的早期跡象。

**偵測策略**：

```
訊號來源矩陣：
┌──────────┬──────────────────────────────┐
│ 主流來源 │ 產業報告、大型平台公告、       │
│          │ 主流媒體報導                  │
│          │ → 通常已是趨勢，非弱訊號      │
├──────────┼──────────────────────────────┤
│ 邊緣來源 │ 技術部落格、Reddit/HN 討論、  │
│          │ 開源專案 star 趨勢、           │
│          │ 小型 conference talks          │
│          │ → 弱訊號的主要來源            │
├──────────┼──────────────────────────────┤
│ 反向來源 │ 失敗案例、被放棄的專案、       │
│          │ 批評文章、反主流觀點            │
│          │ → 「什麼不 work」本身是訊號   │
└──────────┴──────────────────────────────┘
```

**弱訊號強度評估**：

```
訊號強度 = 0.3 × 來源多樣性 + 0.3 × 時間聚集度 + 0.2 × 意外程度 + 0.2 × 連結密度

來源多樣性：多個不相關來源提到同一件事 (0-1)
時間聚集度：短時間內出現頻率突然升高 (0-1)
意外程度：與主流預期的偏離程度 (0-1)
連結密度：能與其他訊號形成的關聯數量 (0-1)

強度 > 0.6 → 強弱訊號，建議升級為 S3 正式追蹤
強度 0.3-0.6 → 中弱訊號，記錄為 Hypothesis 結晶
強度 < 0.3 → 噪音，記錄但不追蹤
```

**報告結構**：

```

    建議行動：在 eval-engine 加入效率維度追蹤

三、可忽略的噪音
  - 量子計算在 AI 的應用（距離實用還有 5+ 年）
  - 歐盟 AI Act 對台灣 SME 的直接影響（極小）

## 掃描執行機制

### 主動掃描

使用者透過指令觸發，Env-Radar 使用 web search 工具即時搜尋最新資訊。

**搜尋策略**：
- 每個維度 2-3 次搜尋，使用精準關鍵字
- 優先搜尋原始來源（官方公告、研究報告、開源專案）
- 交叉比對至少 2 個來源才形成訊號
- 標注每個訊號的來源可靠度和時間新鮮度

### 被動偵測

在日常對話中自然偵測外部環境相關資訊：
- 使用者提到看到的新聞、產業文章 → 擷取為原始訊號
- 使用者分享客戶的新需求或反饋 → 標記為用戶行為信號
- 使用者轉述競品情報 → 錄入競品追蹤

被動偵測到的訊號先入 L0 暫存區，累積到一定量後合併分析。

### 掃描頻率建議

| 維度 | 建議頻率 | 理由 |
|------|---------|------|
| 技術趨勢 | 每週一次 | AI 技術變化快 |
| 市場動態 | 每兩週一次 | 市場週期較長 |
| 競品情報 | 每兩週一次 | 競品動作需累積觀察 |
| 用戶行為 | 持續被動偵測 | 來自日常互動 |
| 政策法規 | 每月一次 | 政策變化慢 |
| 生態系統 | 每月一次 | 平台變化有預告期 |

## 護欄與限制

### 資訊品質護欄
- **來源驗證**：每個訊號必須標注來源，且標明來源可靠度（官方/權威/社群/傳聞）
- **時效標記**：每個訊號標注資訊時間，超過 30 天的資訊降級處理
- **偏見防護**：避免只搜尋確認既有假設的資訊（刻意搜尋反面觀點）
- **噪音過濾**：不是所有新聞都是訊號，分級系統確保只有相關的才上升

### 行動謹慎護欄
- **不因單一訊號做重大決策**：至少 2 個獨立來源交叉確認
- **區分事實與推測**：每個洞見明確標注「已確認」vs「推測中」
- **不製造緊迫感**：除非真的是 S1 緊急，不人為升高壓力等級
- **保持中立**：呈現機會同時也呈現威脅，不偏向樂觀或悲觀

### 倫理護欄
- 競品情報僅來自公開合法來源
- 不模擬競品的未公開策略或內部決策
- 不建議任何可能侵犯智慧財產權的行動
- 政策法規解讀僅供參考，標明「非法律建議」

    執行迭代
         ↓
    eval-engine A/B 比對效果
         ↓
    結果結晶到 knowledge-lattice
         ↓
    Env-Radar 下一輪掃描（驗證方向是否正確）
```

**完整演化閉環**：
- **eval-engine** = 向內看的眼睛（「我變好了嗎？」）
- **env-radar** = 向外看的眼睛（「世界在變什麼？」）
- **morphenix** = 決策引擎（「我該怎麼變？」）
- **knowledge-lattice** = 長期記憶（「我學到了什麼？」）

四者缺一不可，才構成真正的適應性演化。

---

## 哲學總結

> 最好的進化不是隨機變異中碰巧存活，
> 而是先看懂環境壓力的方向，再有目的地變異。
> Env-Radar 讓 MUSEON 不再是「對環境的被動反應者」，
> 而是「對環境的主動偵察者」。
> 但要記住：雷達看到的只是訊號，不是真相。
> 訊號需要被質疑、被驗證、被放在更大的脈絡中才有意義。
> 所以 Env-Radar 永遠標記「確認程度」，永遠保留「反面觀點」。
